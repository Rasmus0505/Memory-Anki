# 09-02 小屏/移动端适配审计与修复

## 1. 原始需求

本项目没有独立移动端，手机经 Tailscale 以 PWA 方式访问**完整桌面前端**（`apps/web/public/manifest.webmanifest` 第 5 行 `"start_url": "/freestyle"`，历史的 `/m`、`/mobile` 路由已退役并在 `apps/web/src/app/router/appRoutes.tsx` 第 118-127 行 `RETIRED_MOBILE_ROUTE_FALLBACKS` 中重定向）。移动端已有的基础设施：`apps/web/src/app/shell/AppShell.tsx` 第 392-433 行的 `MobileBottomNav`（`lg:hidden`、safe-area 底部内边距、包含 freestyle/palaces/review/english/profile 五个入口），以及 `<main>` 的 `pb-[calc(4.5rem+env(safe-area-inset-bottom,0px))] lg:pb-0`（第 500-504 行）为底部导航让位。

但各页面从未做过系统性小屏审计：可能存在固定宽度导致横向滚动、hover 才可见的操作在触屏上不可达、可拖拽/可缩放浮窗（`apps/web/src/shared/components/ui/dialog.tsx` 的 floating 模式，第 36-45 行八向 resize 手柄）在触屏上与页面滚动冲突等问题。本文档产出：核心手机路径的逐页审计清单 + 常见问题的标准修法 + 明确的"不做"边界。

#### 步骤 3：每页检查项清单（viewport 390px）

逐项检查并记录：

1. **横向滚动**：`document.documentElement.scrollWidth > 390` 即为问题。在控制台执行：
   ```js
   [...document.querySelectorAll('*')].filter(e => e.getBoundingClientRect().right > 390 + 1).slice(0, 10)
   ```
   找出溢出元素。
2. **固定宽度**：溢出通常来自 `w-[NNNpx]`、`min-w-[NNNpx]`、`grid-cols-N` 无断点。静态扫描辅助命令（在 `apps/web` 下执行）：
   ```
   rg -n "(?:^|\s)(?:min-)?w-\[(3[6-9][0-9]|[4-9][0-9]{2}|[0-9]{4,})px\]" src/features src/app src/shared/components
   ```
3. **hover 依赖交互**：只在 `hover:` 下出现的按钮/菜单（如 `opacity-0 group-hover:opacity-100`）在触屏不可达。扫描辅助：
   ```
   rg -n "opacity-0 group-hover:opacity-100|invisible group-hover:visible" src/features src/shared/components
   ```
   每处判断：触屏上是否有替代路径（长按/点击展开/始终显示）。
4. **触摸目标尺寸**：主要操作按钮渲染高宽 ≥ 40px（`Button` 默认 `h-9`=36px 可接受，`size="sm"`=32px 的主操作要记录）。
5. **浮窗对话框**：打开该页所有对话框，检查 floating 模式（可拖拽标题栏、八向 resize 手柄）在触屏模拟下是否与滚动冲突、是否超出屏幕、能否关闭。
6. **safe-area**：页面底部固定元素是否被 MobileBottomNav 遮挡（`<main>` 已有全局 padding，但页面内自己 `fixed bottom-0` 的元素要单独检查，例如 FreestylePage 第 1594 行有 `bottom-[max(1rem,env(safe-area-inset-bottom,0px))]` 的浮动条——确认它与底部导航不重叠）。
7. **字体与截断**：长宫殿名/知识点标题是否撑破卡片（应有 `truncate` 或 `line-clamp-*`）。

自查点：六组页面每组都有一张填写完成的审计表（哪怕结论是"无问题"）。

### 阶段 B：修复（标准修法）

> 只修审计表中记录的问题；每修一处，在审计表"修复步骤号"列回填。以下是四类问题的标准修法，给出修改示意——执行时以真实审计结果为准。

#### 步骤 4：固定宽度 → 弹性宽度

修改前（示意）：

```tsx
<div className="w-[480px]">
```

修改后：

```tsx
<div className="w-full max-w-[480px]">
```

网格类：`grid-cols-3` → `grid-cols-1 sm:grid-cols-2 lg:grid-cols-3`（保持 lg 及以上视觉不变——**验证桌面回归**）。

#### 步骤 5：hover 依赖 → 触屏可达

对 `opacity-0 group-hover:opacity-100` 的操作按钮，追加 focus 与触屏兜底：

修改前（示意）：

```tsx
<Button className="opacity-0 group-hover:opacity-100">
```

修改后（`pointer-coarse:` 是 Tailwind v4 内置变体，触屏设备上始终可见）：

```tsx
<Button className="opacity-0 group-hover:opacity-100 focus-visible:opacity-100 pointer-coarse:opacity-100">
```

**不要**为此引入长按手势库。

#### 步骤 6：浮窗对话框触屏降级

打开 `apps/web/src/shared/components/ui/dialog.tsx`。第 179-180 行现状：

```tsx
const resolvedLayout = layout ?? (modal ? 'centered' : 'unstyled')
const floatingEnabled = floating ?? resolvedLayout === 'centered'
```

在文件顶部（第 35 行常量区附近）新增触屏/小屏检测：

```tsx
function isCoarsePointerViewport() {
  if (typeof window === 'undefined') return false
  return window.matchMedia('(pointer: coarse)').matches && window.innerWidth < 1024
}
```

并把 `floatingEnabled` 一行改为：

```tsx
const floatingEnabled = (floating ?? resolvedLayout === 'centered') && !isCoarsePointerViewport()
```

效果：手机上对话框退化为普通居中弹窗（无拖拽、无 resize 手柄、不会被拖出屏幕），桌面行为完全不变。**不要**尝试给八向 resize 手柄做触屏适配。同时确认 `dialog.test.tsx`（`apps/web/src/shared/components/ui/dialog.test.tsx`）仍通过；若测试环境 `matchMedia` 缺失导致报错，在检测函数中已有 `typeof window === 'undefined'` 兜底，jsdom 下 `matchMedia` 需按现有测试惯例 mock——如测试失败，只在测试 setup 补 mock，不改组件逻辑。

#### 步骤 7：被底部导航遮挡的页面内固定元素

修改前（示意）：

```tsx
<div className="fixed bottom-4 right-4">
```

修改后（4.5rem 与 AppShell.tsx 第 502 行 `<main>` 的让位保持一致，lg 以上还原）：

```tsx
<div className="fixed bottom-[calc(4.5rem+env(safe-area-inset-bottom,0px))] right-4 lg:bottom-4">
```

#### 步骤 8：明确不做的事（写给执行者，防止范围膨胀）

- **不做** mindmap 编辑器（`/palaces/:id/edit`、`PalaceEditPage`）的触屏编辑适配。验收标准只有"不崩坏"：页面能打开、能单指拖动画布、不产生 JS 报错、能退出。
- **不做** `/knowledge` 大纲编辑、`/english` 工作台的深度触屏优化（能看、能点即可）。
- **不加**任何移动端专属路由或独立移动版组件（历史 `/m` 路由已退役，不要复活）。
- **不引入** hammer.js 等手势库。
- **不改** `MobileBottomNav` 的五个入口构成（AppShell.tsx 第 394-396 行）。
