# 03-05 AI 生成接口并发与频率约束

## 1. 原始需求

AI 生成类接口目前无任何并发/频率约束。核实现状：

- 题目生成：`apps/api/src/memory_anki/modules/palace_quiz/presentation/router.py` 第 291 行 `POST /palaces/{id}/quiz-generation/images`（async）、第 324 行 `/quiz-generation/text-files`（async）、第 357 行 `/quiz-generation/review-mindmap`（同步 def）、第 378 行 `/chapters/{id}/quiz-generation/outline`（同步 def）、第 257/274 行 `short-answer-feedback`/`explain`（同步 def）；
- AI 拆分：`modules/palaces/presentation/router.py` 第 282 行 `POST /palaces/{id}/editor/ai-split`（同步 def）；
- 英语上传：`modules/english/presentation/router.py` 第 84 行 `POST /english/upload`（async），`task_service.py` 第 62 行 `MAX_UPLOAD_BYTES = 1024 * 1024 * 1024`（1GB）读入内存。

同步 `def` 路由的 LLM 调用靠 starlette 线程池（默认约 40 线程）承载，手机端连点/离线队列重放可能同时触发多个长耗时 LLM 调用与 1GB 级上传，耗尽线程池、内存与付费 API 额度。目标：**进程内**信号量限制同类 AI 任务并发数（超限立刻 429，不排队），外加一个极简滑动窗口频率计数；不引入 Redis/celery/分布式限流（单用户单进程产品）。

## 2. 详细执行清单

### 步骤 1：新建 `core/concurrency_limits.py`

新建文件 `apps/api/src/memory_anki/core/concurrency_limits.py`，完整内容：

```python
from __future__ import annotations

import threading
import time
from collections import deque
from contextlib import contextmanager

from fastapi import HTTPException

# 单用户产品：常量即可，不做运行时配置。
AI_GENERATION_MAX_CONCURRENCY = 2   # 题目生成 / AI 拆分等 LLM 调用
HEAVY_UPLOAD_MAX_CONCURRENCY = 1    # 英语视频上传（1GB 级内存占用）
AI_CALLS_PER_HOUR_LIMIT = 60        # 兜底频率：防离线队列风暴/前端死循环

_SEMAPHORES = {
    "ai_generation": threading.BoundedSemaphore(AI_GENERATION_MAX_CONCURRENCY),
    "heavy_upload": threading.BoundedSemaphore(HEAVY_UPLOAD_MAX_CONCURRENCY),
}

_RATE_LOCK = threading.Lock()
_RATE_WINDOW: deque[float] = deque()
_RATE_WINDOW_SECONDS = 3600.0


def _check_hourly_rate() -> None:
    now = time.monotonic()
    with _RATE_LOCK:
        while _RATE_WINDOW and now - _RATE_WINDOW[0] > _RATE_WINDOW_SECONDS:
            _RATE_WINDOW.popleft()
        if len(_RATE_WINDOW) >= AI_CALLS_PER_HOUR_LIMIT:
            raise HTTPException(
                status_code=429,
                detail="AI 调用过于频繁（每小时上限已达），请稍后再试。",
            )
        _RATE_WINDOW.append(now)


@contextmanager
def concurrency_slot(kind: str, *, rate_limited: bool = False):
    """非阻塞获取并发槽位；拿不到直接 429，绝不排队阻塞线程池。"""
    semaphore = _SEMAPHORES[kind]
    if not semaphore.acquire(blocking=False):
        raise HTTPException(
            status_code=429,
            detail="已有同类 AI 任务在进行中，请等待其完成后重试。",
        )
    try:
        if rate_limited:
            _check_hourly_rate()
        yield
    finally:
        semaphore.release()
```

设计说明：429 而非排队，是因为排队会占住 starlette 线程池线程，恰恰是要避免的；前端 `http.ts` 对 429 不入离线重放队列（只有 ≥500 或冲突才入队，见 `apps/web/src/shared/api/http.ts` 第 213~218 行），用户会直接看到可读错误信息，符合预期。

**自查点**：`cd apps/api && python -c "from memory_anki.core.concurrency_limits import concurrency_slot; print('ok')"` 输出 ok。

### 步骤 5：并发行为测试

新建 `apps/api/tests/test_concurrency_limits.py`：

```python
import threading

import pytest
from fastapi import HTTPException

from memory_anki.core import concurrency_limits
from memory_anki.core.concurrency_limits import concurrency_slot


def test_second_acquire_rejected_while_first_held():
    entered = threading.Event()
    release = threading.Event()

    def hold():
        with concurrency_slot("heavy_upload"):
            entered.set()
            release.wait(timeout=5)

    worker = threading.Thread(target=hold)
    worker.start()
    try:
        assert entered.wait(timeout=5)
        with pytest.raises(HTTPException) as exc_info:
            with concurrency_slot("heavy_upload"):
                pass
        assert exc_info.value.status_code == 429
    finally:
        release.set()
        worker.join(timeout=5)


def test_slot_released_after_exception():
    with pytest.raises(RuntimeError):
        with concurrency_slot("heavy_upload"):
            raise RuntimeError("boom")
    with concurrency_slot("heavy_upload"):
        pass  # 能再次进入说明已释放


def test_hourly_rate_limit(monkeypatch):
    monkeypatch.setattr(concurrency_limits, "AI_CALLS_PER_HOUR_LIMIT", 2)
    concurrency_limits._RATE_WINDOW.clear()
    with concurrency_slot("ai_generation", rate_limited=True):
        pass
    with concurrency_slot("ai_generation", rate_limited=True):
        pass
    with pytest.raises(HTTPException) as exc_info:
        with concurrency_slot("ai_generation", rate_limited=True):
            pass
    assert exc_info.value.status_code == 429
    concurrency_limits._RATE_WINDOW.clear()
```

**自查点**：`cd apps/api && python -m pytest tests/test_concurrency_limits.py -q` 全绿。

### 明确不要做什么

- 不引入 Redis / 分布式限流 / 任务队列框架；进程内 `threading` 原语足够；
- 不把 429 请求排队等待（会占线程池线程）；
- 不改动 `external_ai_call_logs` 记录逻辑与各 service 内部实现；
- 不给普通 CRUD 路由加限制。

可执行命令：

- `cd apps/api && python -m pytest` → 无新增失败（LLM 相关既有测试多为 mock 单次调用，不会触发 429）。

行为验收：

1. 打开两个浏览器标签同时对同一宫殿点"AI 生成题目" → 一个正常执行，另一个立刻收到"已有同类 AI 任务在进行中"的可读报错，不产生第二次 LLM 计费调用（可在 `/api/v1/ai-call-logs` 验证只有一条新记录）。
2. AI 任务完成后重试被拒的请求 → 正常执行。
3. 英语视频上传进行中再次上传 → 429；完成后可再上传。

回归检查：

- 单个 AI 生成/拆分/判分/讲解请求的行为、响应结构与耗时不变；
- 前端离线队列不会把 429 请求入队重放（`http.ts` 仅对 ≥500 与冲突入队）；
- `ai-call-logs` 页面仍完整记录成功与失败调用。
