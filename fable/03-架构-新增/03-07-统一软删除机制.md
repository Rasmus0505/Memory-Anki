# 03-07 统一软删除机制

## 1. 原始需求

目前仅 `study_sessions`（`apps/api/src/memory_anki/infrastructure/db/_tables/misc.py` 第 62 行 `deleted_at`，配套索引第 27 行）与 `mindmap_import_jobs`（同文件第 108 行）有软删除；其余实体全部硬删。其中风险最高的两个：

- 宫殿：`modules/palaces/application/palace_service.py` 第 106~116 行 `delete_palace` 直接删除 DB 行**并物理删除附件文件**，误删一个多年积累的宫殿只能靠备份目录整体恢复；
- 题库题目：`modules/palace_quiz/presentation/router.py` 第 203~224 行支持单删与 `batch-delete` 批量硬删，手机端误触批量删除无法撤销。

软删除价值评估结论：**做** palaces、palace_quiz_questions（误删代价大、恢复诉求真实）；**不做** subjects/chapters（删除有级联确认且结构性强）、attachments（与文件系统强耦合）、review_schedules/review_logs（可由 `repair-stage-progress`/重建服务再生）、segments/mini-palaces（从属宫殿，随宫殿软删已覆盖主要风险）。

目标：alembic 迁移加列 + repository/service 过滤 + 恢复接口的最小方案，30 天后由既有备份体系兜底（本文档不做自动清理任务）。

3b. 打开 `apps/api/src/memory_anki/modules/palaces/application/palace_service.py`，第 106~116 行：

```python
# 修改前
def delete_palace(session: Session, palace_id: int) -> None:
    repo = _repo(session)
    palace = repo.get_palace(palace_id)
    if palace:
        for attachment in palace.attachments:
            filepath = ATTACHMENTS_DIR / attachment.filename
            if filepath.exists():
                filepath.unlink()
        repo.delete(palace)
        repo.commit()

# 修改后：软删，不再物理删除附件文件（附件随最终清理/备份策略处理）
def delete_palace(session: Session, palace_id: int) -> None:
    repo = _repo(session)
    palace = repo.get_palace(palace_id)
    if palace:
        palace.deleted_at = utc_now_naive()
        repo.commit()
```

顶部补 `from memory_anki.core.time import utc_now_naive`。注意：软删宫殿的 `review_schedules` 仍在表里，`GET /review/queue` 通过 `schedule.palace` 关联展示——执行时检查 `modules/reviews/application/review_queue_service.py` 的队列查询，凡 join/引用 Palace 处加 `Palace.deleted_at.is_(None)` 过滤（或在取到 schedule 后跳过 `schedule.palace.deleted_at is not None` 的项），确保软删宫殿不再进复习队列。

3c. 恢复与回收站接口，在 `modules/palaces/presentation/router.py` 的 `api_delete`（第 177 行）附近新增两个路由：

```python
@router.get("/palaces/deleted")
def api_list_deleted(s: Session = Depends(session_dep)):
    rows = (
        s.query(Palace)
        .filter(Palace.deleted_at.isnot(None))
        .order_by(Palace.deleted_at.desc())
        .all()
    )
    return {
        "items": [
            {"id": p.id, "title": p.title, "deleted_at": p.deleted_at.isoformat()}
            for p in rows
        ]
    }


@router.post("/palaces/{palace_id}/restore")
def api_restore(palace_id: int, s: Session = Depends(session_dep)):
    palace = s.query(Palace).filter_by(id=palace_id).first()
    if not palace or palace.deleted_at is None:
        return {"error": "not found"}
    palace.deleted_at = None
    s.commit()
    return {"ok": True, "id": palace.id}
```

重要：`/palaces/deleted` 必须注册在 `GET /palaces/{palace_id}`（第 149 行）**之前**，否则 `deleted` 会被当作 palace_id 解析导致 422。`Palace` 模型在该文件已有 import 渠道（通过 service），直接查询需从 `memory_anki.infrastructure.db.models` import——与文件既有 import 风格一致即可。

**自查点**：删除宫殿后 `GET /palaces` 不含它、`GET /palaces/deleted` 含它、restore 后恢复原状且附件仍可打开。

打开 `apps/api/src/memory_anki/modules/palace_quiz/application/` 下的题目服务（`service.py`，含 `list_questions`/`delete_question`/`batch_delete_questions`）：

```python
@router.post("/palace-quiz-questions/{question_id}/restore")
def api_restore_palace_quiz_question(question_id: int, s: Session = Depends(session_dep)):
    row = s.query(PalaceQuizQuestion).filter_by(id=question_id).first()
    if not row or row.deleted_at is None:

```

### 明确不要做什么

- 不给 subjects/chapters/attachments/review_schedules/segments/mini-palaces 加软删除（评估结论：不值得）；
- 不做定时物理清理任务与"永久删除"接口（备份体系已兜底，回收站数据量可忽略）；
- 不改前端删除交互（恢复入口属后续前端文档；本文档保证 API 可用即可，可先用 curl 恢复）；
- 不动 `study_sessions`/`mindmap_import_jobs` 既有软删实现。
