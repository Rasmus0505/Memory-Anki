# 03-09 扩展幂等机制覆盖全部写接口

## 1. 原始需求

后端幂等机制位于 `apps/api/src/memory_anki/modules/persistence/application/idempotency.py`：以 `X-Memory-Anki-Mutation-ID` 请求头为键，把响应 JSON 存进 `config` 表（key 前缀 `api_mutation.`），重复请求直接返回缓存。grep 全仓核实：目前**仅** `modules/reviews/presentation/router.py` 一处使用（第 187 行 `get_idempotent_response`、第 217 行 `save_idempotent_response`，覆盖 `POST /review/session/{id}/submit` 及其别名 `POST /review/{id}/submit`）。

而前端早已全面就绪：`apps/web/src/shared/api/http.ts` 第 204~206、275~277 行对**所有非 GET 请求**自动注入 mutation id；`apps/web/src/shared/persistence/mutationQueue.ts` 离线重放时复用同一 id。这意味着弱网/Tailscale 断连场景下，队列重放 `POST /study-sessions`、`POST /palaces` 等请求会**重复创建数据**——头发了但后端没接。

目标：按优先级把幂等读写对接到高风险写路由。**前置依赖：02-07（`api_mutation.*` 记录的 TTL 清理）必须先行**，否则每个写请求都会在 config 表永久留一行，表会无限膨胀。

## 2. 详细执行清单

### 统一接入模式（所有步骤复用）

参照 reviews router 第 180~220 行的既有写法，对每个目标路由做四件事：

1. 路由签名加 `request: Request` 参数（`from fastapi import Request` 已有或补充）；
2. 顶部 import：`from memory_anki.modules.persistence.application.idempotency import get_idempotent_response, save_idempotent_response`；
3. handler 第一行：

```python
    existing_response = get_idempotent_response(session, request)
    if existing_response is not None:
        return existing_response
```

4. 在构造好响应 dict、`session.commit()` 之前：

```python
    save_idempotent_response(session, request, response, commit=False)
    session.commit()
    return response
```

若原 handler 由 service 内部 commit（多数 CRUD 如此），采用简化尾部模式——service 调用完成后直接 `save_idempotent_response(session, request, response)`（默认 `commit=True`，多一次 commit 对 SQLite 单用户可接受，避免为此重构 service 的事务边界）。**注意**：响应必须是可 JSON 序列化的 dict（这些路由本来就返回 dict，满足）。

以 `POST /study-sessions` 为例给出完整修改示意：

```python
# 修改前（sessions/presentation/router.py 第 60~65 行）
@router.post("/study-sessions")
def api_create_study_session(data: dict, session: Session = Depends(session_dep)):
    try:
        return {"item": create_study_session(session, data)}
    except ValueError as exc:
        raise HTTPException(status_code=400, detail=str(exc)) from exc

# 修改后
@router.post("/study-sessions")
def api_create_study_session(
    data: dict,
    request: Request,
    session: Session = Depends(session_dep),
):
    existing_response = get_idempotent_response(session, request)
    if existing_response is not None:
        return existing_response
    try:
        response = {"item": create_study_session(session, data)}
    except ValueError as exc:
        raise HTTPException(status_code=400, detail=str(exc)) from exc
    save_idempotent_response(session, request, response)
    return response
```

注意：**只在成功路径缓存**——异常抛出时绝不能 save（否则错误会被"幂等"复现）。sessions router 需在 fastapi import 中补 `Request`。

### 步骤 3：P2 批次——明确**不接入**的路由（防止执行者过度覆盖）

以下路由**天然幂等或低风险，不要接入**（接入只会白白膨胀 config 表）：

- 所有 `PUT`/`PATCH` 全量或合并更新（`PUT /palaces/{id}`、`PUT /settings`、`PATCH /study-sessions/{id}`、各 progress upsert 等）：重放结果相同；
- 所有按 id 的 `DELETE`（`DELETE /palaces/{id}` 等）：重放是 no-op 或返回 not found，无害；
- `POST /study-sessions/bulk-delete`、`POST /palace-quiz-questions/batch-delete`、`/reset-attempts`：按 id 集合操作，天然幂等；
- AI 生成类 `POST`（quiz-generation、ai-split、explain 等）：响应体巨大且 03-05 已做并发约束，缓存大 JSON 进 config 表得不偿失；
- 备份/恢复类 `POST /backups/*`：管理操作，人工触发。

### 步骤 4：幂等行为测试

新建 `apps/api/tests/test_idempotent_writes.py`（TestClient 组织方式参照 `tests/test_study_session_routes.py` 既有写法）：

```python
# 伪代码骨架，client/db fixture 按 tests 既有模式取用
MUTATION_HEADER = {"X-Memory-Anki-Mutation-ID": "test-mutation-0001"}


def test_create_study_session_replay_returns_cached(client):
    payload = {"scene": "review", "target_type": "none", "title": "t"}
    first = client.post("/api/v1/study-sessions", json=payload, headers=MUTATION_HEADER)
    second = client.post("/api/v1/study-sessions", json=payload, headers=MUTATION_HEADER)
    assert first.status_code == 200
    assert second.json() == first.json()
    listing = client.get("/api/v1/study-sessions").json()
    ids = [item["id"] for item in listing["items"]]
    assert len([i for i in ids if i == first.json()["item"]["id"]]) == 1


def test_failed_create_not_cached(client):
    bad = {"scene": ""}  # 触发 400 的载荷，以 create_study_session 校验为准
    headers = {"X-Memory-Anki-Mutation-ID": "test-mutation-0002"}
    assert client.post("/api/v1/study-sessions", json=bad, headers=headers).status_code == 400
    good = {"scene": "review", "target_type": "none", "title": "t2"}
    ok = client.post("/api/v1/study-sessions", json=good, headers=headers)
    assert ok.status_code == 200  # 失败响应未被缓存，同 id 重试成功
```

**自查点**：`cd apps/api && python -m pytest tests/test_idempotent_writes.py -q` 全绿。

### 明确不要做什么

- 不改 `idempotency.py` 本身（存储介质、key 规则、80 字符上限都保持现状；TTL 属 02-07）；
- 不改前端（`http.ts`/`mutationQueue.ts` 已全量发头，无需任何改动）；
- 不给步骤 3 列出的路由接入；
- 不把幂等做成全局中间件（响应缓存必须发生在"成功构造响应之后"，中间件拿不到该时机的事务语义，逐路由显式接入更可控）。

## 3. 测试验收标准

可执行命令：

- `cd apps/api && python -m pytest tests/test_idempotent_writes.py -q` → 全绿。
- `cd apps/api && python -m pytest` → 全部通过（不带 mutation 头的请求行为完全不变——`read_mutation_id` 对缺头请求返回 None，机制整体旁路）。

行为验收：

1. 用同一 `X-Memory-Anki-Mutation-ID` 连发两次 `POST /api/v1/palaces` → 只创建一个宫殿，两次响应体相同。
2. 手机端弱网操作：创建学习会话时断网 → 恢复后 mutationQueue 自动重放 → 会话列表只有一条记录。
3. `sqlite3` 查 `config` 表：`SELECT COUNT(*) FROM config WHERE key LIKE 'api_mutation.%'` → 每个成功写请求一行，且随 02-07 的 TTL 清理回落。
4. 不带头的 curl 写请求（如脚本调用）→ 行为与现状一致，不产生 `api_mutation.` 行。

回归检查：

- `POST /review/session/{id}/submit` 既有幂等行为不变（`test_review_routes.py` 全绿）；
- 失败请求（400/404/500）不产生幂等缓存，修正载荷后同 id 重试可成功；
- 批量创建题目接口的响应（可能较大）写入 config 表后，设置页/其他读 config 的功能不受影响（`read_settings` 只取 DEFAULTS 内键，`api_mutation.*` 前缀键会被一并读出——核实 settings router 第 69~88 行 `read_settings` 是 `for row in session.query(Config).all()` 后 `result[row.key] = row.value`，即 `api_mutation.*` 键**会**混入 `GET /settings` 响应；02-07 清理可缓解，但仍建议在 02-07 或本文档实施时给 `read_settings` 加前缀过滤 `if row.key.startswith("api_mutation."): continue`，实施者择一处落地并在进度表注明）。
