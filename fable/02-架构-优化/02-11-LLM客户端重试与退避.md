# 02-11 LLM 客户端重试与退避

## 1. 原始需求

`apps/api/src/memory_anki/infrastructure/llm/openai_compatible.py`（315 行）是所有 AI 供应商（DashScope/Zhipu/SiliconFlow/DeepSeek）共用的 HTTP 客户端，用 `urllib` 手写，**没有任何重试**：`call_chat_completion_text`（195-237 行）与 `stream_chat_completion_text`（240-267 行）遇到 `urllib.error.URLError`（网络抖动）、HTTP 429（限流）、HTTP 5xx（服务端瞬时故障）都直接抛错。对个人应用而言，一次网络抖动就让一整次出题/导图导入失败，体验差且浪费已消耗的上下文准备工作。

期望效果：对**网络错误、429、5xx** 增加指数退避重试（最多重试 2 次，即总共最多 3 次请求）；**流式请求一旦开始产出数据流就绝不重试**（避免内容重复/错乱），只允许在"连接尚未建立成功"阶段重试；`OpenAICompatibleError` / `OpenAICompatibleHttpError` / `OpenAICompatibleNetworkError` / `OpenAICompatibleProtocolError` 错误层次保持不变，调用方（`ai_service_runtime*` 等）零改动。

## 2. 详细执行清单

> 禁止事项：不要引入第三方重试库（tenacity 等）——urllib 手写风格是本文件的既有选择，保持零依赖；不要对 4xx（除 429）重试（401/403 是配置错误，重试无意义且刷错误日志）；不要改动任何解析函数（`extract_*`、`parse_chat_completion_stream`）；不要改 `OpenAICompatibleChatConfig` 已有字段的默认值。

### 步骤 1：增加重试参数与判定函数

打开 `apps/api/src/memory_anki/infrastructure/llm/openai_compatible.py`。

1. 顶部 import 增加 `import time`。
2. 在 `OpenAICompatibleChatConfig`（14-20 行）中追加两个带默认值的字段（frozen dataclass 追加字段不影响现有构造调用）：

```python
@dataclass(frozen=True, slots=True)
class OpenAICompatibleChatConfig:
    api_key: str
    base_url: str
    model: str
    temperature: float | None = 0.0
    timeout_seconds: float = 90.0
    max_retries: int = 2
    retry_backoff_seconds: float = 1.0
```

3. 在 `_build_network_error`（311 行）之后追加：

```python
def _should_retry_http_status(status_code: int) -> bool:
    return status_code == 429 or 500 <= status_code <= 599


def _sleep_before_retry(attempt: int, base_seconds: float) -> None:
    # attempt 从 0 计：第 1 次重试等 base，第 2 次等 base*2
    time.sleep(base_seconds * (2 ** attempt))
```

自查点：`python -m ruff check src/memory_anki/infrastructure/llm/openai_compatible.py` 通过。

### 步骤 2：非流式请求加重试环

修改 `call_chat_completion_text` 的请求段。

修改前（223-237 行）：

```python
    request_url = build_chat_completions_url(config.base_url)
    request = urllib.request.Request(...)
    try:
        with urllib.request.urlopen(request, timeout=config.timeout_seconds) as response:
            response_body = response.read().decode("utf-8", errors="ignore")
    except urllib.error.HTTPError as exc:
        raise _build_http_error(exc, request_url) from exc
    except urllib.error.URLError as exc:
        raise _build_network_error(exc, request_url) from exc
    return extract_chat_completion_text_from_body(response_body)
```

修改后：

```python
    request_url = build_chat_completions_url(config.base_url)
    last_error: OpenAICompatibleError | None = None
    for attempt in range(config.max_retries + 1):
        request = urllib.request.Request(
            request_url,
            data=json.dumps(payload, ensure_ascii=False).encode("utf-8"),
            headers=_build_headers(config.api_key),
            method="POST",
        )
        try:
            with urllib.request.urlopen(request, timeout=config.timeout_seconds) as response:
                response_body = response.read().decode("utf-8", errors="ignore")
            return extract_chat_completion_text_from_body(response_body)
        except urllib.error.HTTPError as exc:
            last_error = _build_http_error(exc, request_url)
            if not _should_retry_http_status(exc.code):
                raise last_error from exc
        except urllib.error.URLError as exc:
            last_error = _build_network_error(exc, request_url)
        if attempt < config.max_retries:
            _sleep_before_retry(attempt, config.retry_backoff_seconds)
    assert last_error is not None
    raise last_error
```

注意：`Request` 对象在环内重建（urllib 的 Request 可复用，但重建更保险）；读响应体成功后**不再**因解析异常（ProtocolError）重试——协议错误说明模型返回了畸形内容，重试大概率同样结果且会双倍计费。

自查点：写一个临时脚本把 `base_url` 指向不存在的端口（如 `http://127.0.0.1:1`），调用 `call_chat_completion_text` → 观察耗时 ≈ backoff(1s)+backoff(2s)+3 次连接超时，最终抛 `OpenAICompatibleNetworkError`。

### 步骤 3：流式请求只重试"未开始的流"

`stream_chat_completion_text`（240-267 行）的关键约束：一旦 `parse_chat_completion_stream` 开始 `yield`，内容已经流向调用方，绝不能重试。实现方式是把"建立连接 + 拿到 response"与"消费流"分离，只对前者重试：

修改前（261-267 行）：

```python
    try:
        with urllib.request.urlopen(request, timeout=config.timeout_seconds) as response:
            return (yield from parse_chat_completion_stream(response))
    except urllib.error.HTTPError as exc:
        raise _build_http_error(exc, request_url) from exc
    except urllib.error.URLError as exc:
        raise _build_network_error(exc, request_url) from exc
```

修改后：

```python
    last_error: OpenAICompatibleError | None = None
    for attempt in range(config.max_retries + 1):
        request = urllib.request.Request(
            request_url,
            data=json.dumps(payload, ensure_ascii=False).encode("utf-8"),
            headers=_build_headers(config.api_key, accept_sse=True),
            method="POST",
        )
        try:
            response = urllib.request.urlopen(request, timeout=config.timeout_seconds)
        except urllib.error.HTTPError as exc:
            last_error = _build_http_error(exc, request_url)
            if not _should_retry_http_status(exc.code):
                raise last_error from exc
        except urllib.error.URLError as exc:
            last_error = _build_network_error(exc, request_url)
        else:
            # 连接已建立：从这里开始产出数据，任何后续错误一律不重试
            try:
                return (yield from parse_chat_completion_stream(response))
            finally:
                response.close()
        if attempt < config.max_retries:
            _sleep_before_retry(attempt, config.retry_backoff_seconds)
    assert last_error is not None
    raise last_error
```

说明：原实现用 `with` 管理 response，改为手动 `close()`（finally 保证）；流中途的网络中断会由 `parse_chat_completion_stream` 内部的迭代抛错，按设计**不重试**、原样上抛。

自查点：正常配置下前端"AI 文本出题"流式预览工作正常（SSE 逐段出现）；把 base_url 指向坏地址 → 请求重试 2 次后失败，前端收到错误事件。

### 步骤 4：单元测试

在 `apps/api/tests/` 新建 `test_openai_compatible_retry.py`，用 `unittest.mock.patch("urllib.request.urlopen")` 覆盖三个用例：

1. 前两次抛 `urllib.error.URLError("boom")`、第三次返回正常响应 mock → `call_chat_completion_text` 成功且 urlopen 被调 3 次。
2. 抛 `HTTPError(code=401)` → 不重试（urlopen 仅 1 次），抛 `OpenAICompatibleHttpError` 且 `is_auth_error` 为 True。
3. 流式：第一次 urlopen 抛 URLError、第二次返回可迭代 SSE mock → 生成器完整产出；再造一个"流迭代中途抛错"的 mock → 异常上抛且 urlopen 仅 1 次（未重试）。

测试中把 `retry_backoff_seconds` 设为 0 避免真实 sleep。
