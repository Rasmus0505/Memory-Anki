# 02-04 建立全局异常处理器与统一错误响应

## 1. 原始需求

后端目前三种错误风格并存（均已核实）：

1. **HTTPException**（`{"detail": ...}`）：settings、sessions、reviews、freestyle 等 router 的主流写法。
2. **手工 JSONResponse**：`modules/knowledge/presentation/router.py` 第 153 行 `JSONResponse(status_code=409, ...)`、第 166/227/265 行 `JSONResponse(status_code=500, content={"error": tb})`——**把完整 traceback 返回给客户端**。
3. **200 + `{"error": "..."}`**：knowledge router 第 84/97/110/122/134/174 行、`palaces/presentation/router.py` 34 处、`sessions/presentation/router.py` 10 处等，资源不存在时返回 200，前端 `shared/api/http.ts` 的 `request()` 只对非 2xx 抛错，这类响应会被当成正常数据静默穿透。

此外：整个后端**没有注册任何 `app.exception_handler`**（grep `exception_handler` 零结果），未捕获异常直接落到 Starlette 默认 500；`app/main.py` 第 135–138 行 shutdown 备份 `except Exception: pass` 无声吞错。目标：定义统一错误响应 schema，注册全局处理器，修复吞错，并给出各 router 渐进收敛清单。

## 2. 详细执行清单

> 硬约束：分三阶段，阶段 1、2 不改变任何现有接口的状态码；阶段 3 每改一个接口必须先 grep 前端调用方确认影响。不要引入新依赖；不要在本文档里顺手删除 knowledge router 的 print 调试（有独立删减文档处理）。

### 阶段 1：错误 schema 与全局处理器（不改变现有路由行为）

**步骤 1.1**：新建 `apps/api/src/memory_anki/app/error_handlers.py`，完整内容：

```python
"""全局异常处理：统一错误响应为 {"detail": {"code", "message"}}。"""
from __future__ import annotations

import logging

from fastapi import FastAPI, Request
from fastapi.exceptions import RequestValidationError
from fastapi.responses import JSONResponse
from starlette.exceptions import HTTPException as StarletteHTTPException

logger = logging.getLogger(__name__)


def _error_body(code: str, message: str) -> dict:
    return {"detail": {"code": code, "message": message}}


def install_error_handlers(app: FastAPI) -> None:
    @app.exception_handler(StarletteHTTPException)
    async def handle_http_exception(request: Request, exc: StarletteHTTPException):
        # detail 已是 dict 的路由（如 settings ai-models）保持原样，避免破坏前端解析
        if isinstance(exc.detail, dict):
            return JSONResponse(status_code=exc.status_code, content={"detail": exc.detail})
        return JSONResponse(
            status_code=exc.status_code,
            content=_error_body(f"http_{exc.status_code}", str(exc.detail)),
        )

    @app.exception_handler(RequestValidationError)
    async def handle_validation_error(request: Request, exc: RequestValidationError):
        return JSONResponse(
            status_code=422,
            content={"detail": {"code": "validation_error", "message": "请求参数校验失败。",
                                "errors": exc.errors()}},
        )

    @app.exception_handler(Exception)
    async def handle_unexpected_error(request: Request, exc: Exception):
        logger.exception("未处理异常: %s %s", request.method, request.url.path)
        return JSONResponse(
            status_code=500,
            content=_error_body("internal_error", "服务器内部错误，请查看服务端日志。"),
        )
```

**步骤 1.2**：在 `app/main.py` 第 141 行 `app = FastAPI(...)` 之后、`app.add_middleware(...)` 之前加入：

```python
from memory_anki.app.error_handlers import install_error_handlers

install_error_handlers(app)
```

（import 放到文件顶部 import 区。）

**注意前端兼容性**（改动前先确认）：`apps/web/src/shared/api/http.ts` 对非 2xx 用 `extractResponseMessage(status, body)` 提取文案。执行者需先阅读该函数对 `detail` 为字符串/对象两种形态的处理；若它只认 `detail` 字符串，则 `handle_http_exception` 中保持 `detail` 原始形态透传（即字符串 detail 仍返回 `{"detail": "..."}`），把 code 包装放到阶段 3 与前端一起改。**以真实读到的前端代码为准，宁可少包装。**

自查点：启动后端，请求一个不存在的 AI 日志 `GET /api/v1/ai-call-logs/nonexistent` → 404 且响应体可被前端正常 toast（人工在前端触发一次确认）。

### 阶段 3：渐进收敛 200+{"error"} 风格（逐接口、逐个确认前端）

收敛原则：`{"error": "not found"}` → `raise HTTPException(status_code=404, detail="not found")`。**每个接口单独一次改动**，流程固定为：

1. 在 `apps/web/src` grep 该接口路径，找到调用方；确认调用方逻辑——若前端有 `if (res.error)` 之类判断，需同步删除该判断（`request()` 对 404 会直接 throw）；若前端没有判断（多数情况，错误静默穿透），直接改后端即可。
2. 后端替换返回语句。
3. 跑对应路由测试 + 手工触发一次该接口的"不存在"分支。

建议收敛顺序（从测试覆盖最好的开始）：

| 批 | 文件 | 接口 |
|---|---|---|
| 1 | knowledge router | 84/97 (update/delete_subject)、110/122 (get_tree/get_subject_editor)、134、174 (get_chapter)、275、291 |
| 2 | sessions router legacy 进度接口 | 193/201/213/225/245/257/281/293/317/329 行的 `{"error": "not found"}` |
| 3 | palaces router | 34 处（数量大，按资源分 4~5 小批：palace 详情/编辑器/segments/mini-palaces/attachments） |

不要做：不要一批改完所有接口；不要动 `POST /backups/*` 里 `{"error": "missing ..."}` 这类 400 语义的返回（应改为 422/400，放到 02-06 配合 Pydantic 模型一起做，避免重复劳动）。

自查点（每批）：对应测试全绿；前端对应页面的"资源不存在"场景显示错误提示而非白屏/静默。

行为验收：

- 访问不存在资源（如 `GET /api/v1/chapters/999999`，在其所属批次完成后）→ 404 + `{"detail": ...}`，不再是 200。
- 人为制造未捕获异常（临时在某路由 raise RuntimeError）→ 500 + `{"detail": {"code": "internal_error", ...}}`，响应体无 traceback，服务端日志有完整堆栈；验证后还原。
- `PUT /api/v1/subjects/{id}/editor` 传入会触发保存冲突的数据 → 仍是 409（阶段 2 不动该分支）。

回归检查：

- 前端所有页面加载与保存操作正常（重点：学科编辑器保存、章节增删、宫殿列表、复习提交）。
- `tests/test_review_routes.py` 中依赖既有错误文案的断言（如 400"开始日期不能晚于结束日期"）不被破坏——阶段 1 的 HTTPException 包装若影响 `detail` 取值方式，必须回到步骤 1.2 的"透传字符串 detail"方案。
