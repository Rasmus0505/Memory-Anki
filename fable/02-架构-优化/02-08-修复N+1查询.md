# 02-08 修复 N+1 查询

## 1. 原始需求

三处按行数放大的查询热点（SQLite 单文件库，单查询很快，但 ORM 往返次数随数据量线性增长，宫殿/章节多时接口明显变慢）：

1. **面包屑逐级查询**：`apps/api/src/memory_anki/modules/knowledge/presentation/router.py` 169-201 行 `get_chapter`，183-191 行的 while 循环沿 `parent_id` 每层发一条 `s.query(Chapter).filter_by(id=cur_id).first()`——章节深 N 层就是 N 条查询。
2. **章节树递归 lazy load**：同文件 41-52 行 `chapter_json` 递归访问 `c.children` 与 `c.palaces`（`palace_count` 用 `len(c.palaces or [])`），每个节点触发两次 lazy load；`GET /subjects/{id}/tree`（105-115 行）对整棵树调用它，M 个章节 ≈ 2M 条 SQL。
3. **GET /palaces 逐宫殿重序列化**：`modules/palaces/presentation/router.py` 112-114 行 `api_list` 对每个宫殿调 `palace_json(p, s)`；`modules/palaces/application/palace_serializer.py`（全文件 329 行）的 `palace_json`（72 行起）内部每次调用都会执行 `reconcile_palace_chapter_binding`、`get_palace_explicit_chapter_ids`、`palace_stage_progress`、`get_algorithm_stage_labels`、`build_palace_default_segment_summary`、`list_palace_segments`、`list_palace_mini_palaces` 等多个各自带查询的函数——K 个宫殿至少 7K 条查询。

期望效果：三处均改为常数条数（或与层级无关）的查询；响应 JSON 逐字段不变。

## 2. 详细执行清单

> 禁止事项：不要改任何响应 JSON 的字段名/结构/排序语义；不要为了性能改动 `reconcile_palace_chapter_binding` 的写入副作用（它会修正宫殿-章节绑定，列表页依赖它）；不要引入缓存（个人应用，正确性优先）；每处修改前先用步骤 0 的计数器量化，改完复测，无明显下降就回滚该处。

### 步骤 0：搭一个临时查询计数器（验证用，完成后删除）

在测试或临时脚本中用 SQLAlchemy 事件统计 SQL 条数：

```python
from sqlalchemy import event
from memory_anki.infrastructure.db._tables._base import engine

counter = {"n": 0}
event.listen(engine, "before_cursor_execute", lambda *a, **k: counter.update(n=counter["n"] + 1))
```

对目标接口发一次请求，记录 `counter["n"]` 作为基线。

### 步骤 1：面包屑改为一次性查询

`get_chapter`（router 169-201 行；若 02-05 已完成则位于 `modules/knowledge/application/chapter_service.py` 的 `get_chapter_detail`）：

修改前（183-191 行）：

```python
    breadcrumbs: list[dict[str, int | str]] = []
    cur_id = c.parent_id
    while cur_id:
        parent = s.query(Chapter).filter_by(id=cur_id).first()
        if parent:
            breadcrumbs.insert(0, {"id": parent.id, "name": parent.name})
            cur_id = parent.parent_id
        else:
            break
```

修改后（一次查出该学科全部章节的 id/name/parent_id 三列，内存里走链）：

```python
    chapter_index = {
        row.id: row
        for row in s.query(Chapter.id, Chapter.name, Chapter.parent_id)
        .filter(Chapter.subject_id == c.subject_id)
        .all()
    }
    breadcrumbs: list[dict[str, int | str]] = []
    cur_id = c.parent_id
    while cur_id and cur_id in chapter_index:
        parent = chapter_index[cur_id]
        breadcrumbs.insert(0, {"id": parent.id, "name": parent.name})
        cur_id = parent.parent_id
```

自查点：任选一个 3 层以上的章节请求 `GET /api/v1/chapters/{id}`，面包屑顺序与改造前一致；计数器显示该接口 SQL 条数不再随层级增长。

### 步骤 2：章节树 eager load

`get_tree`（router 105-115 行）在取 subject 之外，把整棵树的关系一次性加载。改法：给章节查询挂 `selectinload`。

在 router（或 chapter_service）顶部 import：`from sqlalchemy.orm import selectinload`。

修改 `get_tree`：

修改前：

```python
    subject = s.query(Subject).filter_by(id=subject_id).first()
    if not subject:
        return {"error": "not found"}
    root_chapters = [c for c in subject.chapters if c.parent_id is None]
```

修改后：

```python
    subject = s.query(Subject).filter_by(id=subject_id).first()
    if not subject:
        return {"error": "not found"}
    all_chapters = (
        s.query(Chapter)
        .options(selectinload(Chapter.palaces))
        .filter(Chapter.subject_id == subject_id)
        .all()
    )
    root_chapters = [c for c in all_chapters if c.parent_id is None]
```

说明：`selectinload(Chapter.palaces)` 让 `palace_count` 不再逐节点查询；`children` 关系在同一 session 中因所有行都已加载，SQLAlchemy 仍会按 lazy 逐节点发查询——若计数器显示 children 仍是热点，追加改法：不走 `c.children` 关系，在内存里按 `parent_id` 建 `children_map` 并给 `chapter_json` 增加可选参数 `children_map=None`（为 None 时保持旧行为，别处调用零影响）。

自查点：`GET /api/v1/subjects/{id}/tree` 响应与改造前逐字节一致（可 diff）；SQL 条数从 ~2M 降到 ≤ 5。

### 步骤 3：GET /palaces 列表瘦身

打开 `modules/palaces/application/palace_service.py`，找到 `list_palaces`，给查询追加关系预加载（关系名以 `infrastructure/db/_tables` 中 Palace 模型实际定义为准，执行时先 `rg -n "relationship" src/memory_anki/infrastructure/db/_tables` 核对）：

```python
from sqlalchemy.orm import selectinload

query = query.options(
    selectinload(Palace.review_schedules),
    selectinload(Palace.chapters),
    selectinload(Palace.pegs),
)
```

这消掉 `palace_json` 中 `p.review_schedules`（78 行）、`p.chapters`（285 行）、pegs 的逐宫殿 lazy load。剩余热点是每宫殿的函数调用（stage_progress/segments/mini_palaces 等），它们各自内部有独立查询，动它们属于结构性改造，**本文档只做到 selectinload 这一层**，并在步骤 4 评估是否需要继续。

自查点：`GET /api/v1/palaces` 响应不变；计数器 SQL 条数明显下降（预期从 ~7K+3K 降到 ~4K+常数）。

### 步骤 4：量化收尾与决策记录

用步骤 0 计数器对三个接口记录"改造前/后"SQL 条数，写入本文档进度记录表。若 `GET /palaces` 改造后仍 > 100 条且前端列表页实测仍慢（>500ms），在进度表中记录"建议后续把前端列表切换到 `/palaces/grouped-summary`（palace_summary_json 更轻）或对 palace_json 做批量化改造"，不在本文档内展开。最后**删除步骤 0 的临时计数器代码**。
