# 02-10 修正 CORS 矛盾

## 1. 原始需求

`apps/api/src/memory_anki/app/main.py` 143-149 行：

```python
app.add_middleware(
    CORSMiddleware,
    allow_origins=["*"],
    allow_credentials=True,
    allow_methods=["*"],
    allow_headers=["*"],
)
```

按 Fetch 规范，`Access-Control-Allow-Origin: *` 与 `Access-Control-Allow-Credentials: true` 互斥。Starlette 的 CORSMiddleware 对这种组合的实际行为是回显请求 Origin——等价于"任意来源都可带凭据跨域"，是最宽松、最反规范的形态。

实际访问面核实（决定正确配置的关键事实）：

1. **生产/日常**：FastAPI 在 `main.py` 167-168 行直接 mount 前端产物（`WEB_DIST_DIR`），浏览器访问 `http://127.0.0.1:8012` 或 Tailscale HTTPS 域名时，页面与 API **同源**，根本不产生 CORS 请求。
2. **开发**：`apps/web/vite.config.ts` 206-214 行配置了 dev server 代理——`/api` → `http://127.0.0.1:8012`，浏览器只与 `localhost:5173` 通信，同样**同源**。
3. 项目不使用 Cookie/Session 凭据（无任何认证，纯个人内网应用），`allow_credentials=True` 没有存在理由。

结论：CORS 中间件在当前访问面下几乎不被触发，但保留一个宽松兜底可以容忍"偶尔从别的端口/工具直接调 API"的场景。期望效果：改为规范合法且默认收紧的配置——显式来源清单 + 关闭凭据。

## 2. 详细执行清单

> 禁止事项：不要引入新的配置文件或复杂的来源解析框架（一个环境变量足够）；不要顺手改动同区域的 `RequestLoggingMiddleware`（150 行）与缓存头中间件（151 行）；不要把 Tailscale 域名硬编码进代码（跨设备约束：设备名不同域名不同）。

### 步骤 1：定义来源清单（环境变量可扩展）

打开 `apps/api/src/memory_anki/app/main.py`，在 141 行 `app = FastAPI(...)` 之前新增：

```python
def _resolve_cors_origins() -> list[str]:
    """默认覆盖本机后端与 Vite dev；Tailscale HTTPS 域名等额外来源经环境变量追加。"""
    origins = [
        "http://127.0.0.1:8012",
        "http://localhost:8012",
        "http://127.0.0.1:5173",
        "http://localhost:5173",
    ]
    extra = os.environ.get("MEMORY_ANKI_CORS_ORIGINS", "")
    origins.extend(o.strip() for o in extra.split(",") if o.strip())
    return origins
```

顶部补 `import os`（当前 main.py 没有 import os，需确认后添加）。

说明：同源访问（127.0.0.1:8012、Tailscale 域名走同一个后端）本不需要 CORS，把它们列进来只是防御性冗余；`5173` 覆盖开发时绕过代理直连后端的少数场景（如手写 fetch 绝对地址调试）。

### 步骤 2：替换中间件配置

修改前（143-149 行）：

```python
app.add_middleware(
    CORSMiddleware,
    allow_origins=["*"],
    allow_credentials=True,
    allow_methods=["*"],
    allow_headers=["*"],
)
```

修改后：

```python
app.add_middleware(
    CORSMiddleware,
    allow_origins=_resolve_cors_origins(),
    allow_credentials=False,
    allow_methods=["*"],
    allow_headers=["*"],
)
```

`allow_headers=["*"]` 保留：前端自定义头 `X-Memory-Anki-Mutation-ID`（幂等机制，见 `modules/persistence/application/idempotency.py` 12 行）需要被预检放行。

自查点：`python -m ruff check src/memory_anki/app/main.py` 通过。

### 步骤 3：在两台设备的启动配置中登记 Tailscale 域名（如需要）

若确有从"非本机源"跨域直调 API 的用法（执行者先向使用者确认；默认没有），在启动脚本/服务配置中为每台设备设置：

```
MEMORY_ANKI_CORS_ORIGINS=https://<本机的-tailscale-域名>
```

没有该用法则跳过本步骤，什么都不用配。
