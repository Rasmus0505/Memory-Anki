# 05-07 英语听力任务流水线可靠性

## 1. 原始需求

英语听力课程生成（上传视频 → 提取音轨 → ASR 转写 → 整理句子 → 翻译 → 保存课程）跑在**模块级单例 + 本地守护线程**上，重启的代价过高：

1. **任务状态在表里，但执行在内存线程里**。`apps/api/src/memory_anki/modules/english/application/task_service.py`（628 行）的 `_runtime` 是模块级单例（第 74-78 行），`LocalThreadEnglishTaskRunner`（`infrastructure/task_runner.py` 第 13-21 行）用守护线程跑 `run_generation_task`（task_service.py 第 320-484 行）。任务行持久化在 `english_generation_tasks` 表（`create_task_row` 第 262-317 行），阶段进度也一直写库——**持久化地基已存在**。
2. **但启动清理会"毁尸灭迹"**。`cleanup_incomplete_generation_tasks`（第 611-627 行，经 `modules/english/application/startup.py` 的 `prepare_english_runtime` 在启动时调用）把所有非 `completed/cleared` 的任务直接置为 `cleared`，并 `shutil.rmtree` 删掉任务目录——**包括用户上传的原始视频**（`source{suffix}`，第 274-275 行）。也就是说：任务跑到翻译阶段时重启，用户不但丢了进度，连重试的原料都被删了，只能重新上传 1GB 以内的视频。
3. **重试不复用已完成的昂贵步骤**。`retry_current_task`（第 135-154 行）只允许 `failed` 状态重试，且新任务从头跑：音轨重新提取（便宜）、**ASR 重新转写（最贵、最慢）**、翻译重新做。ASR 结果目前只进转写流程，没有落盘为工件。

期望效果：重启后未完成任务变成"已中断（可重试）"而不是被清空删档；重试时若上一次任务的 ASR 结果已落盘则直接复用，只重跑后续阶段。**不做**外部任务队列、进程池、多任务并发（`create_generation_task` 第 112-116 行的"同时只有一个任务"约束保持不变）。

### 流水线阶段与重试成本（核实自 `run_generation_task`）

| 阶段 | stage 值 | 成本 | 现有落盘产物 |
|---|---|---|---|
| 提取音轨 | `extract_audio` | 低（本地 ffmpeg） | `audio.wav` |
| ASR 转写 | `transcribe` | **高（云端 ASR，长音频分钟级）** | 无 ← 本文档补上 |
| 整理句子 | `prepare_sentences` | 低（纯本地） | 无 |
| 翻译 | `translate` | 中（LLM 调用） | 无 |
| 保存课程 | `finalize` | 低 | 课程入库后删任务目录 |

## 2. 详细执行清单

> 硬性约束：**不要**改 `EnglishRuntime` 单例与 `configure_english_runtime` 的测试注入机制；**不要**改 SSE 流 `stream_task_events`（第 202-259 行）的事件协议；**不要**引入新表（`english_generation_tasks` 已够用）；**不要**支持多任务并发。

#### 步骤 1：启动清理改为"标记中断、保留现场"

打开 `apps/api/src/memory_anki/modules/english/application/task_service.py`，替换 `cleanup_incomplete_generation_tasks`（第 611-627 行）：

```python
# 修改后：queued/running → failed（可重试，保留任务目录）；其余历史垃圾照旧清掉
def cleanup_incomplete_generation_tasks(session: Session) -> dict[str, int]:
    interrupted = 0
    cleared = 0
    tasks = (
        session.query(EnglishGenerationTask)
        .filter(EnglishGenerationTask.status.notin_(tuple(STARTUP_PRESERVED_TASK_STATUSES)))
        .all()
    )
    for task in tasks:
        if task.status in ACTIVE_TASK_STATUSES:
            # 执行线程已随进程消失，标记为失败态以复用既有重试入口；保留任务目录
            task.status = "failed"
            task.stage = "interrupted"
            task.message = "生成因服务重启被中断，可点击重试继续。"
            task.error_message = "服务重启导致任务中断。"
            task.updated_at = utc_now_naive()
            interrupted += 1
            continue
        if task.status == "failed":
            # 已失败任务保留现场，等用户手动重试或清除
            continue
        task.status = "cleared"
        task.stage = "cleared"
        task.message = "启动时清理历史任务。"
        task.updated_at = utc_now_naive()
        shutil.rmtree(task_dir(task.id), ignore_errors=True)
        cleared += 1
    if interrupted or cleared:
        session.commit()
    return {"cleared": cleared, "interrupted": interrupted}
```

说明：置为 `failed` 而不是新造 `interrupted` 状态，是为了让既有的 `retry_current_task`（只认 `failed`）、`clear_current_task`（认 `failed/queued`）和前端"重试/清除"按钮**零改动**直接可用。`stage="interrupted"` 用于区分展示。`prepare_english_runtime`（startup.py 第 8-9 行）返回值多了 `interrupted` 键，调用方（`startup_runtime.py` 第 90 行）不读返回值，无需改。

自查点：`cd apps/api && python -m pytest tests/test_english_routes.py -x` 通过（若既有用例断言了 `{"cleared": N}` 的精确结构，同步更新断言加上 `interrupted` 键）。

#### 步骤 3：重试时把 ASR 工件带到新任务目录

同文件 `retry_current_task`（第 135-154 行），在 `create_task_row(...)` 之后、`task.status = "retried"` 之前插入：


## 3. 测试验收标准

**可执行命令：**

```bash
cd apps/api && python -m pytest tests/test_english_routes.py -x    # 期望全部通过（含 2 个新增用例）
cd apps/web && npm run typecheck                                    # 期望 0 错误
cd apps/web && npm run test -- EnglishWorkspacePage                 # 期望全部通过
curl http://127.0.0.1:8012/api/v1/english/current-task
# 重启后期望：{"status":"failed","stage":"interrupted","message":"生成因服务重启被中断，可点击重试继续。",...}
```

**行为验收（操作 → 期望现象）：**

1. 上传视频开始生成，在"正在识别字幕"阶段强杀后端并重启 → 英语工作台显示"生成被服务重启中断"+ 重试按钮，任务目录与源视频仍在磁盘上。
2. 点重试 → 任务重新执行；若中断发生在 ASR 完成之后（`asr_result.json` 已存在），日志面板出现"复用已完成的 ASR 转写结果"，且总耗时明显缩短。
3. 中断发生在 ASR 完成之前 → 重试正常重跑 ASR，流程与改动前一致。
4. 点"清除"失败任务 → 任务目录被删除，工作台回到可上传状态（既有行为不变）。
5. 任务正常完成 → 任务目录照旧被 `finalize_generation_task` 清理（第 524 行），不残留大文件。

**回归检查：**

- SSE 流 `/english/tasks/{id}/stream` 的 status/log/done/error 事件协议不变。
- "同时只允许一个任务"约束不变（存在 failed 任务时上传会提示先重试或清除，第 113-116 行）。
- 课程完成后的落库（`finalize_course_from_task`）、课程列表、播放进度均不受影响。
- 磁盘占用：failed 任务目录被保留是**有意为之**，但用户点"清除"后必须删干净（`clear_current_task` 第 169 行既有逻辑）。
