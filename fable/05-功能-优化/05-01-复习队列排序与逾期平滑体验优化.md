# 05-01 复习队列排序与逾期平滑体验优化

## 1. 原始需求

当前复习队列的排序与逾期平滑存在三个已核实的问题：

1. **逾期项没有优先级**。`apps/api/src/memory_anki/modules/reviews/application/review_queue_service.py` 中 `_due_query`（第 28-56 行）的排序是 `order_by(ReviewSchedule.review_number, ReviewSchedule.id)`，逾期 10 天的项和今天刚到期的项混排，逾期久的宫殿不会排在队列前面，容易被每日上限（`daily_max_reviews`）截断而永远轮不到。
2. **平滑操作是"黑盒"**。`spread_overdue`（同文件第 181-218 行）直接改库并 commit，`POST /review/spread-overdue`（`modules/reviews/presentation/router.py` 第 128-131 行）只返回 `{"ok": true, "spread": count}`；且前端没有任何地方手动调用它（已全仓搜索确认），用户唯一入口是自动平滑 `maybe_auto_smooth_overdue`（第 221-232 行），在 `GET /review/queue` 时静默触发。
3. **无法撤销**。自动平滑发生后前端 `apps/web/src/app/router/review/ReviewOverview.tsx`（第 125-132 行）仅显示一条"系统已将 N 项逾期任务自动平滑"的横幅，用户既看不到哪些项被移动了，也无法撤销。

期望效果：逾期越久的宫殿越优先出现在队列里；用户可以在复习总览页手动触发平滑，触发前能预览影响范围，触发后能一键撤销（含自动平滑）。

#### 步骤 2：给 `spread_overdue` 增加 dry-run 与移动明细返回

同文件，重写 `spread_overdue`（第 181-218 行）。保持函数签名向后兼容（原调用方 `maybe_auto_smooth_overdue` 与 router 都只传 `days`），新增 `dry_run` 参数，返回值从 `int` 改为 `dict`：

```python
# 修改后（骨架，候选筛选逻辑第 183-201 行原样保留）
def spread_overdue(session: Session, days: int = 7, dry_run: bool = False) -> dict:
    ...  # 原有 candidates / overdue 筛选保持不变
    if not overdue or days <= 0:
        return {"count": 0, "moves": []}

    per_day = max(1, len(overdue) // days)
    moves: list[dict] = []
    for index, schedule in enumerate(overdue):
        offset = index // per_day
        next_date = today + timedelta(days=min(offset, days - 1))
        previous_due_at = (
            schedule_display_datetime(schedule, schedule.palace, session)
            if schedule.palace
            else None
        )
        moves.append({
            "schedule_id": schedule.id,
            "palace_id": schedule.palace_id,
            "palace_title": schedule.palace.title if schedule.palace else "",
            "old_date": schedule.scheduled_date.isoformat(),
            "old_at": schedule.scheduled_at.isoformat() if schedule.scheduled_at else None,
            "new_date": next_date.isoformat(),
        })
        if not dry_run:
            schedule.scheduled_date = next_date
            if previous_due_at is not None:
                schedule.scheduled_at = datetime.combine(next_date, previous_due_at.time())
    if not dry_run:
        _save_spread_undo_snapshot(session, moves)
        session.commit()
    return {"count": len(moves), "moves": moves}
```

同时修改两处调用方以适配新返回值：
- 本文件 `maybe_auto_smooth_overdue` 最后一行 `return spread_overdue(session, days)` 改为 `return spread_overdue(session, days)["count"]`；
- `modules/reviews/presentation/router.py` 的 `api_spread` 见步骤 4。

自查点：`python -m pytest tests/test_review_routes.py -x` 通过（该文件第 301-343 行有自动平滑用例，覆盖 `maybe_auto_smooth_overdue` 的返回值类型）。

#### 步骤 3：撤销快照（存 Config 表，不新增表）

同文件顶部补充 `import json`，在 `spread_overdue` 上方新增两个函数：

```python
_SPREAD_UNDO_KEY = "overdue_spread_undo_snapshot"

def _save_spread_undo_snapshot(session: Session, moves: list[dict]) -> None:
    from memory_anki.infrastructure.db.models import Config

    row = session.query(Config).filter_by(key=_SPREAD_UNDO_KEY).first()
    payload = json.dumps({"created_at": datetime.now().isoformat(), "moves": moves})
    if row:
        row.value = payload
    else:
        session.add(Config(key=_SPREAD_UNDO_KEY, value=payload))


def undo_spread_overdue(session: Session) -> int:
    from memory_anki.infrastructure.db.models import Config

    row = session.query(Config).filter_by(key=_SPREAD_UNDO_KEY).first()
    if not row or not row.value:
        return 0
    snapshot = json.loads(row.value)
    restored = 0
    for move in snapshot.get("moves", []):
        schedule = session.query(ReviewSchedule).filter_by(id=move["schedule_id"]).first()
        if schedule is None or schedule.completed:
            continue  # 已完成的复习不回滚，避免覆盖用户操作
        schedule.scheduled_date = date.fromisoformat(move["old_date"])
        schedule.scheduled_at = (
            datetime.fromisoformat(move["old_at"]) if move.get("old_at") else None
        )
        restored += 1
    row.value = ""
    session.commit()
    return restored
```

注意：`Config` 模型定义在 `apps/api/src/memory_anki/infrastructure/db/_tables/misc.py` 第 140-146 行（key 唯一、value 为 Text），本仓库已有多处用它做 KV 存储（如 `modules/persistence/application/idempotency.py`），照此风格即可。**不要**建新表或写 alembic 迁移。

自查点：Python REPL 中对测试库先 `spread_overdue(session, 7)` 再 `undo_spread_overdue(session)`，确认 scheduled_date 恢复原值且返回数量一致。

**行为验收（操作 → 期望现象）：**

1. 造 3 个宫殿：A 逾期 10 天、B 逾期 2 天、C 今天到期 → 打开 `/review`，队列顺序为 A、B、C。
2. 设置 `daily_max_reviews=2` → 队列只显示 A、B（逾期久的不被截断）。
3. 关闭自动平滑（设置页"自动平滑"开关关掉），存在逾期项 → 复习总览页标题区出现"平滑逾期（N）"按钮。
4. 点击"平滑逾期" → 出现确认卡片，列出前 5 条"宫殿名: 旧日期 → 新日期"；点"取消"数据库无变化。
5. 点"确认平滑" → 队列刷新，横幅显示平滑数量并带"撤销"按钮。
6. 点"撤销" → toast 提示恢复数量，逾期项回到原日期，队列恢复原状。
7. 平滑后先完成其中一项复习，再点"撤销" → 已完成项不被回滚，其余恢复。

**回归检查：**

- 自动平滑（`auto_smooth_overdue=true` 且逾期数达阈值）仍在打开队列时自动执行，`smoothed_count` 横幅照常显示。
- `POST /review/spread-overdue` 不带 `dry_run` 时行为与响应中 `spread` 字段与原版兼容。
- 复习提交（`POST /review/session/{id}/submit`）、章节队列（`/review/chapter/{id}/queue`）不受影响。
