# 05-02 复习阶段进度修复流程自动化

## 1. 原始需求

复习阶段进度数据可能因中断、跨设备同步或历史 bug 变脏（孤儿进度、阶段序列断档），目前修复完全靠用户"知道并主动去点按钮"：

1. **修复入口埋得深**。`POST /review/repair-stage-progress`（`apps/api/src/memory_anki/modules/reviews/presentation/router.py` 第 134-137 行）唯一的前端入口是设置页 `apps/web/src/features/profile/ProfileSettingsPage.tsx` 的 `handleRepairReviewStageProgress`（第 112-142 行），普通使用路径不会经过它。
2. **启动时不会自动修复也不会自检**。一次性迁移 `run_review_schedule_repair_migration`（`apps/api/src/memory_anki/app/startup_runtime.py` 第 63-73 行）只在 prepare 模式（`run_prepare_runtime` 第 96 行）执行，且靠 `REVIEW_SCHEDULE_REPAIR_MIGRATION_KEY` 标记后**永远只跑一次**；serve 模式的 `initialize_service_runtime`（第 109-130 行）和 `main.py` 的 `lifespan`（第 117-138 行）都不会再检查数据是否变脏。之后新产生的脏数据没有任何机制发现。
3. **用户看不到"需要修复"的信号**。`repair_review_stage_progress`（`modules/reviews/application/review_execution_service.py` 第 149-162 行）内部有四类修复动作（孤儿进度迁移第 271-309 行、孤儿学习会话迁移第 312-360 行、练习进度恢复第 363-401 行、进度同步第 404-466 行），但没有任何只读的"体检"函数可以告诉用户当前存在多少问题。

期望效果：后端提供一个**只读**的健康检查接口；服务启动时在既有 warmup 线程里自检并记日志；前端复习总览页发现异常时显示横幅，用户一键触发既有修复接口。**不做**启动时自动执行修复——`repair_review_stage_progress` 会调用 `rebuild_all_pending_review_schedules` 全量重建所有宫殿的排程（`schedule_rebuild_service.py` 第 272-312 行），静默改数据风险太大，必须由用户确认后触发。

### 后端部分

#### 步骤 1：新增只读检测函数

打开 `apps/api/src/memory_anki/modules/reviews/application/review_execution_service.py`，在 `repair_review_stage_progress`（第 149 行）上方新增一个函数。它只做 SELECT，不做任何写操作：

```python
def detect_review_stage_progress_issues(session: Session) -> dict:
    """只读体检：统计需要 repair_review_stage_progress 处理的脏数据数量。"""
    schedule_ids = {row[0] for row in session.query(ReviewSchedule.id).all()}

    orphan_progress_count = sum(
        1
        for progress in session.query(SessionProgress)
        .filter(SessionProgress.session_kind == "review", SessionProgress.completed == False)
        .all()
        if progress.review_schedule_id is None
        or progress.review_schedule_id not in schedule_ids
    )

    orphan_study_session_count = sum(
        1
        for row in session.query(StudySession)
        .filter(
            StudySession.scene == "review",
            StudySession.target_type == "review_schedule",
            StudySession.status.in_(ACTIVE_STATUSES),
            StudySession.deleted_at.is_(None),
        )
        .all()
        if row.target_id is None or row.target_id not in schedule_ids
    )

    # 阶段断档：同一宫殿里存在"未完成阶段的编号 < 已完成阶段的编号"
    stage_gap_palace_count = 0
    schedules_by_palace: dict[int, list[ReviewSchedule]] = {}
    for schedule in session.query(ReviewSchedule).all():
        schedules_by_palace.setdefault(schedule.palace_id, []).append(schedule)
    for items in schedules_by_palace.values():
        pending = [item.review_number for item in items if not item.completed]
        completed = [item.review_number for item in items if item.completed]
        if pending and completed and min(pending) < max(completed):
            stage_gap_palace_count += 1

    total = orphan_progress_count + orphan_study_session_count + stage_gap_palace_count
    return {
        "orphan_progress_count": orphan_progress_count,
        "orphan_study_session_count": orphan_study_session_count,
        "stage_gap_palace_count": stage_gap_palace_count,
        "total_issues": total,
        "needs_repair": total > 0,
    }
```

注意：`ReviewSchedule`、`SessionProgress`、`StudySession`、`ACTIVE_STATUSES` 在本文件顶部已 import（孤儿判定逻辑与第 280-283 行、第 326-329 行的既有写法保持一致），不需要新增 import。**不要**在这个函数里调用任何 `_migrate_*` / `_recover_*` / `_sync_*` 函数。

**行为验收（操作 → 期望现象）：**

1. 干净数据库启动前后端，打开 `/review` → 无警告横幅，网络面板可见 `stage-progress-health` 返回 `needs_repair: false`。
2. 用 SQL 手工制造一条孤儿进度（`INSERT INTO session_progress (session_kind, completed, review_schedule_id) VALUES ('review', 0, 999999)`）→ 刷新 `/review`，出现"检测到 1 处复习进度异常"横幅。
3. 点击"一键修复" → toast 提示"修复完成：重建 N 个宫殿"，横幅消失，队列刷新。
4. 重启后端并观察日志 → warmup 完成前若仍有脏数据，日志出现 `review stage progress self-check found ... issue(s)` 警告行。
5. 设置页原"修复历史宫殿复习进度"按钮 → 行为与改动前一致。

**回归检查：**

- `POST /review/repair-stage-progress` 的响应结构不变（`ProfileSettingsPage.test.tsx` 第 54 行起的用例通过）。
- prepare 模式一次性迁移（`test_review_routes.py` 第 2974-3010 行两个用例）不受影响。
- 复习队列加载（`GET /review/queue`）耗时无明显变化（体检请求是独立并行请求，不阻塞队列渲染）。
