# 01-05 删除退化的多算法抽象

## 1. 原始需求

历史上系统支持多种复习算法，现只剩艾宾浩斯一种，但"算法可选"的抽象壳仍在到处传参。核心证据：

- `apps/api/src/memory_anki/modules/reviews/application/schedule_policy.py` 第 27-28 行：

```python
def normalize_algorithm(algorithm: str | None) -> str:
    return "ebbinghaus"
```

  无条件返回常量，入参完全无意义；
- 同文件 `get_algorithm_intervals_for_policy(policy, algorithm)`（第 65-72 行）、`get_initial_same_day_slot_count_for_policy(policy, algorithm)`（第 75-86 行）的 `algorithm` 参数在函数体内从未被使用；
- `schedule_rebuild_service.py` 第 65-71 行 `segment_algorithm()` 恒返 `"ebbinghaus"`，第 315-316 行 `_default_algorithm()` 恒返 `"ebbinghaus"`，第 109/275 行的 `algorithm_override` 参数收下后被忽略（函数体第 112/282 行直接写死 `algorithm = "ebbinghaus"`）；
- 大量调用方仍在构造 `next((normalize_algorithm(x.algorithm_used) ...), "ebbinghaus")` 这类必然得到 `"ebbinghaus"` 的表达式（如 `reviews/presentation/router.py` 第 68-75 行、`palace_serializer.py` 第 94-101 与 194-201 行）。

**必须保留不动的**：数据库列 `review_schedules.algorithm_used`（alembic 迁移 `0009_restore_review_schedule_algorithm_used.py` 专门恢复过该列）、ORM 字段、`ReviewScheduleDraft.algorithm_used`、API JSON 输出里的 `"algorithm_used"` 键。本文档只删"参数传递与判断"，写库时一律写常量 `"ebbinghaus"`。
