# 文档模板（所有 fable 设计文档必须遵守）

> 本模板是 fable 文件夹下所有改进建议文档的强制格式。执行 AI 模型请严格按照文档结构逐节阅读并执行。

## 元信息（每个文档头部必须有）

```markdown
---
编号: XX-YY（分类编号-文档序号，如 02-03）
标题: 一句话说明本文档要做什么
类型: 删减 | 优化 | 新增
范围: 架构 | 功能 | 前端
优先级: P0（必须） | P1（应该） | P2（可以）
预估工作量: S（<2h） | M（2-8h） | L（>8h）
依赖文档: 无 | [编号列表]
状态: 未开始 | 进行中 | 已完成 | 已阻塞
负责代理: 无 | subagent 标识
完成时间: 无 | YYYY-MM-DD HH:mm
---
```

## 正文结构（必须包含以下四节）

### 1. 原始需求

- 用 2-5 句话说明为什么要做这件事（现状的问题 + 期望达到的效果）。
- 必须引用真实文件路径和现状证据（如"`apps/web/src/xxx.tsx` 目前有 800 行"）。

### 2. 详细执行清单

- 有序步骤列表，每一步都要具体到"打开哪个文件、找到什么代码、改成什么样"。
- 面向能力较弱的 AI 编程模型撰写：
  - 每一步只做一件事；
  - 给出完整文件路径（相对仓库根目录，如 `apps/web/src/features/review/...`）；
  - 涉及代码修改时给出修改前后的代码示意（可省略无关行）；
  - 明确"不要做什么"（避免模型顺手改坏其他文件）；
  - 涉及新文件时给出完整文件内容或逐段说明。
- 步骤末尾附"自查点"：执行者做完后如何立即验证这一步没做错。

### 3. 测试验收标准

- 可执行的验证命令（如 `cd apps/web && npm run typecheck`）与期望结果。
- 行为验收：用"操作 → 期望现象"的格式列出人工验证步骤。
- 回归检查：明确哪些既有功能不能被破坏。

### 4. 进度记录（交接用）

| 时间 | 执行者 | 动作 | 结果/备注 |
|---|---|---|---|
| - | - | 文档创建 | - |

> 执行代理每完成一个阶段必须在此表追加一行；接手的代理先读此表再动手。

## 硬性约束（来自项目根目录 1.md 与 AGENT.md）

1. 最小改动：清单只包含达成目标必需的步骤，禁止顺手重构无关代码。
2. 外科手术式修改：每个改动行都必须能追溯到本文档的原始需求。
3. 跨设备可用：不得写死本机路径；运行时数据位置一律经 `memory_anki.core.config` 派生。
4. 架构边界：前端 `shared` 不依赖 `app/features/entities`；后端 domain 不感知 FastAPI/SQLAlchemy。
5. 验证驱动：每个清单必须有可执行的成功判据，做完必须跑对应检查命令。
