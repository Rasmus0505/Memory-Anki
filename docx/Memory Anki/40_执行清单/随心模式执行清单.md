# 随心模式（刷抖音式学习）执行清单

> 基于代码调研（2026-07-02）撰写，可直接对照文件开工。范围：宫殿浮窗可拖拽/滑出、查看/翻卡模式切换、刷题范围配置、不中断刷题进度/计时。

## 重要发现：两项能力已经基本做完，不是从零开始

调研结果和最初设想不同——"宫殿浮窗可拖拽/滑出"和"查看/翻卡模式切换"这两项能力，**已经在 `PalaceQuizMemoryLookupDialog` 组件里实现，并且 freestyle 页面已经在用它**（`FreestylePage.tsx` 里"查看宫殿"按钮打开的就是这个弹窗）。真正需要开工的核心缺口是**刷题范围配置里"按完整宫殿刷完"的分组粒度不对**，以及对已有能力做一次针对随心场景的验收走查。

---

## 任务 1：修正"按完整宫殿刷完再换下一个"的分组粒度

**现状 -> 目标对照**
- 现状：`buildFreestyleQueue`（`apps/web/src/features/freestyle/model/freestyle.ts`）在 `orderMode === 'palace_complete_then_random'` 时，按 `card.group_key` 分组洗牌，而 `group_key` 的粒度是 `mini:{id}` 或 `palace:{id}`——如果一个宫殿下有多个 mini_palace，各 mini_palace 分组会被打散穿插其他宫殿的内容，不满足"刷完一个完整宫殿、再进入下一个宫殿"的字面语义。
- 目标：分组键统一提升到宫殿级别（`palace_context.id`），保证同一宫殿的所有题目（不论属于哪个 mini_palace）在队列里连续出现，刷完再进入下一个宫殿。

**具体改动**
- 文件：`apps/web/src/features/freestyle/model/freestyle.ts`（`buildFreestyleQueue` 函数）
- 把分组逻辑从读 `card.group_key` 改为读 `card.palace_context.id`（或等价字段，需要先确认 `quiz_cards.py::build_quiz_cards` 序列化出的题卡对象里宫殿 id 字段名，参考 `apps/api/src/memory_anki/modules/freestyle/application/card_context.py`）。
- 如果前端拿到的题卡数据结构里没有直接暴露宫殿级 id（只有 `group_key` 字符串），可能需要后端 `quiz_cards.py` 额外多吐出一个 `palace_id` 字段，或者前端从 `group_key` 里解析 `mini:` 前缀对应的 mini_palace 再反查其所属宫殿——需要看 `card_context.py` 里 mini_palace 到 palace 的映射关系是否已经在序列化数据里带出来。

**验收标准**
- 设置刷题范围为"随机全部"、顺序为"完整宫殿刷完再随机"时，实际做题顺序里同一宫殿的所有题目（跨 mini_palace）连续出现，不会中途插入其他宫殿的题。
- 现有测试 `apps/web/src/features/freestyle/model/freestyle.test.ts` 需要新增一条覆盖"同宫殿多 mini_palace 时顺序不被打散"的用例。

## 任务 2：刷题范围/顺序配置文案与语义打磨（小改动）

**现状 -> 目标对照**
- 现状：`FreestyleSettingsDialog` 已经提供范围（全部/待复习/需练习/指定宫殿）和顺序（`palace_complete_then_random`/`random`/`sequential`）两个独立配置项，后端 `feed_service.py` 已支持 `FREESTYLE_RANGE_ALL/DUE/NEEDS_PRACTICE/SPECIFIC_PALACES` 四种范围过滤。整体功能已经具备，只是产品笔记里的表述（"类别：完整宫殿/随机"、"范围：随机/待复习/指定宫殿"）和代码现状（多了一个"需练习"范围、顺序里多了一个"sequential"）有些出入。
- 目标：核对现有的"范围×顺序"两个维度是否已经完全覆盖你的实际需求，是否需要合并/精简某个选项，让配置面板的用词和选项数量对齐真实使用习惯。

**具体改动**
- 文件：`apps/web/src/features/freestyle/FreestyleSettingsDialog.tsx`（或相近命名，需在该目录下确认具体文件名）
- 这一步不涉及底层代码结构变化，只是产品决策：确认要不要保留"需练习宫殿"这个范围、"sequential"这个顺序选项，是否要改文案让用户更容易理解"范围"和"顺序"两个维度的区别。

**验收标准**
- 产品所有者本人试用一遍配置面板，确认所有范围/顺序组合的实际行为符合预期，文案不产生歧义。

## 任务 3：验收走查已实现的浮窗拖拽/滑出能力

**现状 -> 目标对照**
- 现状：`PalaceQuizMemoryLookupDialog.tsx` + `memoryLookupLayout.ts`（`apps/web/src/features/palace-quiz/`）已实现拖拽、部分滑出屏幕边缘（保留 44px 可见边缘，`MEMORY_LOOKUP_VISIBLE_EDGE`）、拖回、折叠为胶囊等能力，用原生 PointerEvent 手写实现（未用第三方拖拽库），比计时器悬浮窗（目前完全限制在视口内、只能折叠不能真正滑出）更完整。
- 目标：不需要重新开发，只需要针对随心模式场景做一次专项验收，确认这套已有能力在 freestyle 页面里的表现符合预期。

**验收标准**
- 在随心模式做题过程中打开宫殿浮窗，拖到屏幕边缘能部分滑出并保留可点击的边缘用于拖回；拖拽/折叠/展开过程中当前刷题进度（`currentIndex`）、正确连击（`correctStreak`）、计时器（`useTimedSession`）均不受影响、不重置。
- 多分辨率/多显示器场景下走查一遍 `clampMemoryLookupLayoutToViewport` 的边界计算是否有异常（比如窗口大小变化后浮窗残留在屏幕外无法拖回）。

## 任务 4：验收走查已实现的查看/翻卡模式切换

**现状 -> 目标对照**
- 现状：`PalaceQuizMemoryLookupDialog.tsx` 第 545-578 行已实现"查看模式/翻卡模式"切换按钮，翻卡模式复用 `useRevealSession`（`apps/web/src/entities/review/model/useRevealSession.ts`）和 `review-flow-tree.ts` 的节点揭示状态机，通过 `MindMapFrame` 的 `onNodeClick` 接入，未新增任何后端接口，完全符合最初的设计要求。
- 目标：同样不需要重新开发，做一次针对随心模式的专项验收。

**验收标准**
- 在随心模式刷题过程中打开宫殿浮窗并切换到翻卡模式，点击节点触发揭示，不影响当前刷题的 `progress`/`timer` 状态（这两者在代码结构上已经完全独立，`revealSession` 的 state 只存在于弹窗内部）。
- 关闭浮窗或切换回查看模式后，回到刷题页面能无缝继续，不需要重新加载或丢失当前进度。

## 任务 5：不中断刷题进度/计时——针对新增配置项的收尾检查

**现状 -> 目标对照**
- 现状：`settingsOpen`/`todaySettingsOpen`/`memoryLookupOpen` 都是独立的 `useState`，不触碰 `timer`（`useTimedSession` + `useGlobalTimerRegistration`，`persistKey: 'freestyle'`）和 `progress`（`localStorage` key `memory-anki.freestyle.progress`）；切换 `range`/`orderMode` 时 `queueSignature` 变化会 clamp `currentIndex` 但不清零。这个机制本身是对的。
- 目标：如果任务 1、2 里对配置项做了改动（比如新增/合并选项），确保新配置改动复用同一套 `setConfigAndPersist` + `queueSignature` 机制，不引入新的独立状态导致进度意外丢失。

**验收标准**
- 修改刷题范围/顺序配置后，`currentIndex` 按 `queueSignature` 变化正确重新对齐（不越界、不清零），已完成的做题记录（`resolvedQuestionIds`）在同一 signature 下继续生效。
- 全链路手动测一遍：开始刷题 → 中途打开宫殿浮窗查看/翻卡 → 关闭浮窗 → 中途修改范围配置 → 确认计时器和进度符合预期反应（该保留的保留，该按新范围重新排队的重新排队）。
