# 反馈系统重建执行清单

> 基于代码调研撰写，可直接对照文件开工。范围：配置页布局+后端生效问题、烟花五种类型当前预览、烟花触发位置统一为边缘喷发、新增微观即时反馈层（覆盖所有有对错判定的场景）。

## 关键结论：产品所有者反馈的"配置不生效"问题，其实已经被上一轮重构修复/移除了

调研发现一个重要情况：commit `9438c34`（2026-06-20，"重建个人中心反馈配置，烟花类型点击即预览"）已经把旧版"形容词强度档位"（轻提示/平衡/庆祝/电影感）整条链路（`FeedbackSceneIntensity`、`scenes.*.intensity`、`revealFxIntensity`、`criticalFxIntensity`）删除了，改为"视觉强度由 5 种烟花类型本身内置"。当前 `reviewFeedbackSettings.ts` 里已经没有这些形容词字段，测试 `ProfileFeedbackPage.test.tsx:117-119` 明确断言"轻提示""电影感"文案已被移除。

**这意味着：如果你现在打开应用还能看到"轻体""电影感"这类选项，说明你看到的是旧版本或未刷新的构建，需要先确认能否在最新代码上复现问题**，再决定后续工作量。下面的任务清单已经按"最新代码上依然存在的真实缺口"来组织。

## 任务 1：烟花五种类型"点击即预览"——已完成，仅需验收

**现状**：`ProfileFeedbackSettingsPanel.tsx` 的 `ConfettiSegment`（第63-104行）点击按钮时已经直接调用 `launchCelebrationPreset({ preset, reducedMotion, amount: 1.15, scenario: 'preview' })`，这是 `celebrationEngine.ts` 导出的真实触发函数，会在当前整个页面（全局共享 canvas）真实播放该类型烟花，**不是走独立小画布模拟**，也不依赖右侧"实时预览台"。

**验收标准**
- 逐一点击五种类型（random direction/realistic/fireworks/stars/school pride），每次点击后 1 秒内在当前页面看到对应类型的真实烟花效果。
- 如果点击后没有反应，最可能的原因是 `canEmitConfetti(reducedMotion)`（`ProfileFeedbackPage.tsx` 第432行附近）因为某些环境判断（比如系统开启了"减少动态效果"、或被误判为 `prefers-reduced-motion`）提前 return 导致预览被跳过——需要检查当前系统/浏览器的无障碍设置状态。

**说明**：右侧"实时预览台"（`ProfileFeedbackPage.tsx` 第295-432行）现在职责已经收窄，只服务于完成结算/答题/计时器等**非烟花类**场景的功能性演示（依赖 `useRevealSession`），和烟花类型预览不冲突、不重复，不需要改动。

## 任务 2：烟花触发位置统一为"从视口边缘喷出、最高层级"——需要实际开发

**现状（`celebrationEngine.ts`）**
- Z-index 已经是最高（`GLOBAL_CONFETTI_CANVAS_ID` 的 `style.zIndex = '2147483647'`，`position: fixed; inset: 0`），这一项**已达标，不需要改**。
- 喷发位置目前不统一，大部分预设仍偏中心/局部区域：
  - `random_direction`：origin x/y 在 0.08~0.92 随机，偏画面内部。
  - `realistic_look`：固定 `origin.y ≈ 0.68`，画面下方偏中。
  - `fireworks`：origin x 在 0.1~0.3 / 0.7~0.9，y 在 -0.18~0.28，左右两侧但仍在视口内，非贴边。
  - `stars`：origin 固定 `{x:0.06,y:0.84}` 和 `{x:0.94,y:0.84}`，已接近左右下角，较贴边。
  - `school_pride`：`origin.x` 固定 0 和 1（左右边缘），`angle: 60/120` 朝内喷——**这是目前唯一真正"贴边"的预设**，可作为其他预设改造的参照模板。

**具体改动**
- 文件：`apps/web/src/shared/feedback/celebrationEngine.ts`
- 逐个调整 `random_direction`、`realistic_look`、`fireworks` 三个预设的 `emitBurst` 调用参数：
  - 把 `origin.x` 固定改为 0 或 1（或 0.02/0.98 留一点边距），不再用随机内部坐标。
  - `origin.y` 改用画面上边缘（0）或下边缘（1），不用画面中段的固定值。
  - 相应调整 `angle` 让粒子朝画面内喷发：左边缘出发用 `angle ≈ 45-70`，右边缘用 `angle ≈ 110-135`，顶部出发用 `angle ≈ 250-290`（参照 `school_pride` 现有的 60/120 角度设计）。
- `stars` 预设已经比较接近边缘，只需要把 0.06/0.94 微调到 0.02/0.98 让视觉上更贴边（可选，不是必须）。

**验收标准**
- 五种类型触发时，粒子的发射原点都在视口边缘（上/左/右边缘），不再有任何一种从画面中央或内部区域喷出。
- 每种类型改造后手动过一遍视觉效果，确认朝向内侧喷发的角度自然、不会出现粒子刚出现就飞出视口消失的情况。
- 建议改完后逐一在任务1的预览按钮里点一遍确认效果符合预期，这也是任务1和任务2之间天然的联动验收方式。

## 任务 3：微观即时反馈层——新增能力，覆盖所有有对错判定的场景

**现状**
- 已有 `GlobalFeedbackProvider.tsx` 的 `emitDescriptor` 机制：对 `level: 'action'` 事件生成一个 DOM burst（CSS 动画 `memory-anki-feedback-burst 0.62s`，420-620ms TTL）。
- `quiz_result_correct`/`quiz_result_incorrect` 已经在 `globalFeedbackProfiles.ts`（第104-121行）注册为该 level，`quizResultFeedback.ts` 里通过 `dispatchGlobalFeedback` 触发——即测验答题的即时反馈**已经有一个基础通路存在**，但时长是 420-620ms，比目标的"200ms 内、安静、极快"要重一些，且需要确认这套反馈在视觉上是否和里程碑级的彩带庆祝有足够区分度。

**具体改动**
- 文件：`apps/web/src/shared/feedback/globalFeedbackModel.ts`、`globalFeedbackProfiles.ts`、相关 CSS（`memory-anki-feedback-burst` 动画定义所在文件，需要在 `shared/feedback` 或全局样式文件里定位）。
- 第一步：把 `getBurstTtlMs` 或对应的 CSS 动画时长从当前的 420-620ms 收紧到目标的 200ms 区间，用于所有 `level: 'action'` 的判分类事件。
- 第二步：**盘点所有"有对错判定"的场景**，不只是测验选择题，确认这些场景是否都已经接入 `globalFeedbackProfiles.ts` 的 action level 通路：
  - 测验选择题（`quiz_result_correct`/`quiz_result_incorrect`）——已接入。
  - 随心模式刷题的对错判定——需要确认 `features/freestyle` 里的答题反馈是否复用了同一套 `dispatchGlobalFeedback`，还是有自己独立的反馈实现（如果是后者需要统一接入）。
  - 简答题 AI 评分反馈——需要确认是否也算"对错判定"场景，如果算，需要检查当前是否有类似的即时反馈还是只有 toast 提示。
  - 英语听力打字校验、英语阅读校准——如果这些场景也有明确的"对/错"判定，同样需要检查是否接入统一反馈通路。
- 第三步：确保这套"微观即时反馈"在视觉设计上（颜色、图标、动效强度）明显区别于连击里程碑/完成庆祝类的"宏观反馈"，避免用户对两者的重要性感到混淆。

**验收标准**
- 所有已识别的"有对错判定"场景，答对/答错都能在 200ms 内看到轻量视觉反馈。
- 微观反馈和宏观里程碑反馈（彩带、连击音效）在视觉强度上有清晰区分，不会让用户误以为普通一次答对触发了"大庆祝"。
- 新增/统一接入的场景补充对应的前端测试用例（参考 `apps/web/src/shared/feedback/` 目录下现有测试文件的模式）。
