# 体验层清单

> 独立审视结论。聚焦前端美观、交互体验、学习反馈（视觉/听觉）、断点续跑、进度保存、意外情况保存、减少直觉疑惑、UI/UX。每项标注优先级并给出"现状 -> 目标"对照。

---

## 一、视觉与听觉反馈

### [P0] 反馈系统重建：配置要真实生效、要可预览
- **现状**：个人中心反馈配置板块本身"未达标"——形容词类调整（轻体、电影感等）修改后不生效；音量设计表现良好；烟花特效（canvas-confetti）已定义 random direction / realistic / fireworks / stars / school pride 五种由轻到重的类型，但前端缺少直接预览，用户点击某个类型时看不到它的真实效果，只能依赖右侧一个独立的实时预览台；烟花触发位置也没有统一，应该固定从视口边缘喷出、显示层级最高，而不是屏幕中央或局部区域弹出。
- **目标**：重建这个配置页的布局和后端联动逻辑——点击任意类型立即在当前界面（不是另一个预览区）触发一次真实效果；所有配置修改持久化且刷新后仍然生效；统一烟花从视口边缘发射、层级高于所有其他 UI 元素。
- **理由（Why）**：反馈系统是"高反馈学习"这个产品承诺最直接的载体；如果连配置本身都不可信、不直观，用户会对整个反馈体系失去信任，这比反馈效果本身不够炫更严重。
- **验收标准**：五种烟花类型逐一点击都能在当前页面立即看到对应效果；所有反馈相关设置修改后重进应用仍然保持；烟花视觉表现统一为"从边缘喷出、最高层级"。

### [P1] 区分"微观即时反馈"与"宏观里程碑反馈"
- **现状**：现有彩带庆祝、连击里程碑、TTS 播报体系已经比较完善，但主要服务"宏观里程碑"（连击 5 次、进度过半、全部完成）。测验答对/答错这种高频、瞬时的判分动作，目前和宏观庆祝共用同一套"庆祝引擎"心智，没有专门为它设计的"安静、极快、不打断节奏"的反馈层。
- **目标**：为测验类的高频判分动作设计一套独立的、200ms 内完成的轻量视觉反馈（颜色/图标/微动效），和里程碑级的大庆祝在设计语言上明确区分——小反馈要"快而不抢戏"，大庆祝要"稀疏而有分量"。
- **理由（Why）**：如果每道题的对错都触发和"完成全部复习"一样级别的视觉刺激，用户会很快对庆祝麻木，真正的里程碑时刻反而失去特殊感；反馈的层级感本身就是心流体验的一部分。
- **验收标准**：测验答对/答错有独立的轻量反馈组件，响应时间在 200ms 以内，视觉强度明显低于连击/完成类庆祝；两套反馈用不同的组件实现，不共用同一个"庆祝引擎"入口。

## 二、心流与沉浸体验

### [P0] 随心模式：宫殿速览浮窗与翻卡切换
- **现状**：你已明确设计——随心模式下应能不打断当前刷题状态呼出宫殿速览浮窗，浮窗要能像计时器悬浮窗一样部分滑出屏幕、可拖拽回边缘；浮窗右侧脑图标题旁应有"查看模式/翻卡模式"切换，翻卡模式复用现有复习脑图的节点揭示逻辑（不需要新增后端接口）。目前这套设计尚未落地。
- **目标**：按上述设计实现，做题时随手呼出宫殿浮窗查看上下文，不中断当前刷题进度和计时。
- **理由（Why）**：这是"刷抖音式学习"体验里最关键的一环——你需要在高强度连续刷题时，随时能"回头看一眼这道题所在的宫殿全貌"而不被迫退出刷题流程，否则连续性和沉浸感会被打断。
- **验收标准**：随心模式做题界面可以呼出宫殿浮窗，浮窗可拖拽、可部分滑出屏幕边缘再拖回；浮窗内可以在"查看/翻卡"两种模式间切换，翻卡模式行为与正式复习脑图一致；呼出浮窗期间当前刷题计时和进度不受影响。

### [P1] 全屏"沉浸模式"，用减少界面元素而非增加动效来奖励专注
- **现状**：当前没有一个"连续正确/连续专注后自动收起多余界面元素"的状态；界面元素密度在任何时候都相对固定。
- **目标**：设计一个沉浸模式状态——进入连续正确/连续专注一段时间后，自动收起导航、次要按钮等非必要元素，只留核心卡片/题目居中显示，计时器降到最小存在感，支持纯键盘操作。
- **理由（Why）**：更容易进入心流的方式往往是"减少摩擦"而不是"堆叠更多刺激"；持续加庆祝特效边际效益会递减，但持续减少界面噪音的边际效益是稳定的。
- **验收标准**：满足触发条件（如连续答对 N 题）后界面自动进入精简状态，非核心元素隐藏或收起；用户可以随时手动退出沉浸模式恢复完整界面；纯键盘操作路径在沉浸模式下可用。

### [P1] 全局悬浮计时器覆盖"离开产品也要提醒"场景
- **现状**：`desktop-timer` 已经是能在主窗口最小化时依旧置顶悬浮的 Electron 窗口，但场景设计局限在"学习/休息切换"，没有覆盖"我要去做完全无关的事情（比如看电视）时，也希望这个计时器能帮我设个提醒，到点叫我回来"这个真实需求。
- **目标**：扩展成"通用离场提醒"——可以在离开产品前设定一个自定义时长，计时器在你做其他任何事情时依然可见或能主动提醒（系统通知/声音），到点后引导你回到学习状态。
- **理由（Why）**：你已经有了系统级悬浮窗和跨应用置顶的技术基础，这是低成本、高实用性的场景扩展，直接服务"不打断生活节奏地管理学习-休息边界"这个真实痛点。
- **验收标准**：可以设置一个自定义时长的"离场倒计时"；倒计时结束时即便当前焦点在其他任意应用（包括全屏视频播放器），也能收到明显的视觉或声音提醒。

## 三、断点续跑 / 进度保存

### [P0] 修复页面状态丢失与宫殿编辑页打不开的阻断性 bug
- **现状**：路由驻留缓存本应保留页面状态，但实测从书架打开宫殿、切到其他侧边栏页面再切回，看到的是书架而非刚才打开的宫殿；这个问题很可能不止这一处触发路径；同时宫殿编辑页存在直接打不开的问题。
- **目标**：全面排查驻留缓存在带参数路由（如 `/palaces/:id`）下的状态保持逻辑，确保切走再切回精确恢复到离开时的具体状态；修复宫殿编辑页无法打开的问题；对 AI 生成等异步任务也要验证切页后任务和显示内容是否保留。
- **理由（Why）**：这是当前最频繁触发、最直接破坏信任感的体验缺陷——"号称能保留状态但实际上随时可能丢"，比完全不做这个机制更让人困惑，因为行为不可预测，用户无法建立稳定预期。
- **验收标准**：书架→宫殿详情→切走→切回，必须精确停留在宫殿详情；宫殿编辑页在所有已知路径下都能正常打开；覆盖至少一条端到端回归测试防止再次退化。

### [P2] AI 生成任务的断点续跑
- **现状**：`SessionProgress` 层面的断点续跑（做题/复习到一半）已经做得扎实，每次操作同步 commit 落盘；但 PDF 转题库这类耗时的 AI 生成任务如果中途中断，目前需要重新生成，没有断点恢复能力。
- **目标**：把"操作中断后能从断点恢复"这个保证延伸到 AI 生成任务上，中途失败或应用意外关闭后，能从已完成的部分继续，而不是从零重新生成。
- **理由（Why）**：AI 生成任务往往耗时较长（尤其 PDF/图片批量识别），重新生成不仅浪费时间还浪费 AI 调用成本，断点续跑对这类任务的价值和对做题/复习进度的价值同样重要。
- **验收标准**：一个多阶段/多页的 AI 生成任务中途中断后重新进入，能看到已完成部分并从断点继续，而不是重新触发整个流程。

## 四、意外情况处理与数据安全感

### [P1] 离线队列状态对用户可见
- **现状**：`mutationQueue.ts`（IndexedDB 离线/失败重试队列）在网络异常或服务端冲突时自动入队，每 30 秒轮询重放，但完全没有面向用户的状态展示——用户无法知道"我刚才的修改到底存上了没有"，只能靠刷新页面赌运气。
- **目标**：增加一个持久的小状态指示器（如已同步/保存中/离线排队中/有冲突待处理），让这个隐性风险变成可感知、可信任的状态。
- **理由（Why）**：这是当前"意外情况可见性"方面最薄弱的一环——数据层面其实已经有重试机制兜底，但用户体验层面完全没有对应的信任信号，容易造成"我是不是又要重做一遍"的焦虑。
- **验收标准**：界面上任意时刻能看到当前同步状态；发生冲突或长时间未同步成功时有明确提示，并提供手动重试入口。

### [P1] 全局回收站，覆盖当前硬删除的实体
- **现状**：只有宫殿脑图有版本历史可以回溯；题目、章节、附件、迷你宫殿等一旦删除即为硬删除，没有找回渠道。
- **目标**：为高频编辑的实体增加"最近删除"的软删除+限时可恢复能力（与架构层清单的数据层条目对应）。
- **理由（Why）**：个人产品没有人替你兜底，一次手滑就是真实数据损失；这个安全网的实现成本远低于它能避免的心理负担。
- **验收标准**：删除题目/章节/附件/迷你宫殿后能在"最近删除"里找到并一键恢复，超过保留期后再清理。

## 五、减少直觉疑惑 / UI 一致性

### [P2] 统一确认弹窗方案
- **现状**：`native-dialog.tsx` 的 `appConfirm`/`appAlert` 已是主流方案，但还有一个平行的 `confirm-dialog.tsx` 仅在测验管理页使用，导致同样的删除类操作在不同页面弹窗长得不一样。
- **目标**：把最后这一处统一迁移，只保留一套确认弹窗方案。
- **理由（Why）**：视觉/交互不一致是最容易在潜意识层面造成"这个操作是不是不一样、会不会有不同后果"的疑惑来源，即便功能上没有实质差异。
- **验收标准**：全站删除类操作使用同一套确认弹窗组件，视觉表现完全一致。

### [P2] 明确"什么场景用 toast、什么用弹窗、什么用屏幕反馈"的内部规范
- **现状**：toast、确认弹窗、屏幕像素反馈、彩带庆祝各自有自己的适用场景，但目前更多是历史形成的默契，没有写成明确规则。
- **目标**：写一份简短的内部交互规范（哪怕只是几行说明），明确不同严重程度/不同性质的操作分别对应哪种反馈形式，作为未来新功能的默认参照。
- **理由（Why）**：即使当前实现基本合理，缺少书面规范意味着未来新增功能容易各自发明一套，逐渐重新引入不一致；这份规范的写作成本很低，但能长期防止体验碎片化。
- **验收标准**：产出一份简短文档，覆盖"删除类/保存类/生成完成类/连击里程碑类/网络异常类"分别对应的反馈形式；新功能开发时能直接查阅参照。
