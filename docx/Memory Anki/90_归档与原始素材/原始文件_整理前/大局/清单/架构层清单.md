# 架构层清单

> 独立审视结论。回答"这个项目未来应该长成什么架构"，覆盖前端、后端、数据、AI 工作流、同步、测试与可维护性。每项标注优先级并给出"现状 -> 目标"对照。

---

## 一、同步架构（分量最重的一项）

### [P0] 生产/稳定版分离与数据库共享
- **现状**：只在两台设备之间用百度网盘做整包快照同步，没有"生产开发环境"和"日常稳定使用环境"的隔离；你已经明确提出用 GitHub Desktop 新增一个 worktree 专门做开发，现有目录留作稳定学习环境的设想，但数据库如何共享/隔离尚未定案。
- **目标**：建立 worktree 级别的代码隔离（生产开发 worktree + 稳定使用目录），数据库单独存放在一个不随 worktree 切换的共享位置（例如固定在 `MEMORY_ANKI_HOME` 之外或之内的独立子目录，两个 worktree 的运行时都指向同一路径），使得：开发中的新功能验证通过后可以直接让稳定端切换使用同一份数据；如果生产端出问题，可以一边修复一边用稳定版继续正常学习，两边操作的是同一套数据不会产生数据分叉。
- **理由（Why）**：这是你已经提出、也是本项目"个人自用但要频繁自己改代码"这个特殊场景下的刚需——普通产品不需要在"自己是唯一开发者也是唯一用户"之间反复横跳，但这个项目必须解决"改代码不能打断学习"和"学习数据不能因为切分支/切目录而分裂"这两个目标的冲突。
- **验收标准**：两个 worktree（或目录）运行时通过同一个 `MEMORY_ANKI_HOME`（或显式配置的共享数据库路径）读写同一份 SQLite 数据库；在开发 worktree 里改代码验证不影响稳定目录当前运行的进程；两者切换使用不需要手动搬运或转换数据库文件。
- **注意点**：需要确认 SQLite WAL 模式下两个不同代码版本的进程是否可能同时写同一个数据库文件——如果两个 worktree 会同时运行（一边开发一边使用），需要明确"同一时刻只允许一个进程持有数据库写锁"的机制，避免两个不同 schema 版本的进程互相踩踏（尤其是数据库迁移不向后兼容时）。

### [P0] 从"整包快照同步"走向"可解释的合并"
- **现状**：`file_sync.py` 实现的是启动拉取/关闭推送的整包快照同步，本质要求严格串行使用两台设备；一旦两边都在未同步的情况下产生修改，直接阻断启动，需要人工去 `conflicts/` 目录处理，且不会自动合并。
- **目标**（分两步，先做轻量改良，观察是否够用，再决定是否值得做架构级重做）：
  1. **轻量改良**：保留整包快照思路，但把"冲突即阻断、翻文件夹"升级为应用内的冲突处理向导，直接展示两份快照在关键数据上的差异，引导选择保留哪一份或哪些部分（对应功能层清单里的同类条目）。
  2. **架构级重做（可选，视投入意愿）**：把 SQLite 的写操作视为可重放的操作日志/事件流，两台设备互相拉取、按实体粒度做真正合并，而不是整包覆盖；可以是一个极简的自建同步端点。这一步工作量显著更大，只有在"确实频繁遇到冲突、轻量改良不够用"时才值得投入。
- **理由（Why）**：当前方案能用但脆弱，风险集中在"必须记得按顺序开关应用"这一操作纪律上，一旦破坏就要人工救火；轻量改良能用较低成本大幅降低救火频率和救火难度。
- **验收标准**：轻量改良阶段——冲突发生时应用内能展示可读的差异和处理选项，不需要打开文件管理器；重做阶段（如启动）——两台设备可以不严格串行使用，各自产生的修改能自动合并而非互相覆盖。

## 二、后端架构

### [P1] 收敛英语听力/阅读到统一会话与内容源基础设施
- **现状**：`english`、`english_reading` 各自维护平行的 session、progress、AI 生成任务实现，和宫殿/复习体系完全独立。
- **目标**：抽出一个通用的"学习会话"和"内容源"抽象，英语听力（音视频源）、英语阅读（分级文本源）作为这个抽象上的两种内容适配器接入，而不是独立重造。
- **理由（Why）**：三份平行实现意味着未来任何"进度保存、断点续跑、事件记录"层面的改进都要在三个地方各做一次；收敛后新内容类型（比如未来想加"视频课程"之外的其他媒介）也能直接复用。
- **验收标准**：英语听力/阅读的会话进度落库结构与宫殿复习共用同一张核心表（或至少共用同一套 CRUD 服务契约），修改一处断点续跑逻辑能同时惠及三类内容。

### [P1] 清偿已登记的架构边界豁免
- **现状**：`docs/architecture/boundary-exceptions.json` 里登记了 4 条"临时"跨模块依赖（reviews↔palaces 两条、backups↔mindmap 两条），均已写明移除条件但尚未清偿。
- **目标**：把这 4 条豁免当作一个正式的技术债清偿任务排期处理，而不是让"临时"状态无限展期。
- **理由（Why）**：反复展期的临时豁免就是事实上的永久债务；这个项目本身有很规范的架构治理机制（豁免登记表、`check_architecture.py`），不清偿等于让治理机制形同虚设。
- **验收标准**：4 条豁免逐条按登记的移除条件实现（例如抽出 `ReviewTargetPort`），并从 `boundary-exceptions.json` 中删除对应条目，配套回归测试通过。

### [P2] `palace_quiz` 模块重新归并
- **现状**：`application` 层被拆成约 150 个小文件，架构规则要求不许经 `service.py` 门面互引，是全仓库文件粒度最细的模块。
- **目标**：按内聚的命令簇（如"生成""批改""分类""导入导出"）重新归并文件，降低认知负担，同时保留清晰的职责边界。
- **理由（Why）**：过度拆分和过度耦合一样是可维护性问题——找一个功能要跳十个文件，本身就是认知成本。
- **验收标准**：文件数量显著下降（比如归并到 20-40 个按命令簇组织的文件），单文件职责仍然清晰，`check_architecture.py` 检查通过。

## 三、调度算法

### [P2] 从艾宾浩斯/SM2 混合调度迁移到 FSRS
- **现状**：`reviews` 模块用艾宾浩斯/SM2 混合策略，固定间隔表驱动复习调度。
- **目标**：迁移到 FSRS（Free Spaced Repetition Scheduler）算法，根据你真实的复习历史（正确率、耗时、连续失败等）反推个性化遗忘曲线参数。
- **理由（Why）**：FSRS 是目前公认优于 SM2/朴素艾宾浩斯的开源调度算法，调度越准，"该复习的时候恰好推给你"这件事的信噪比越高，这是"高反馈"承诺的算法基础，也是遗忘曲线预测（功能层清单）能落地的前提。
- **验收标准**：新调度算法上线后，能对比旧算法在"复习负担/遗忘率"上的实际差异（哪怕只是主观感受记录一段时间）；旧的 SM2 数据能平滑迁移或至少不丢失历史记录。

## 四、AI 工作流

### [P1] 统一 AI 网关层
- **现状**：思维导图导入、测验生成、ASR、CEFR 分级等每个场景各自硬编码调用哪家供应商、什么模型，各自维护自己的 Job 表和轮询逻辑；`external_ai_call_logs` 目前只是调用日志。
- **目标**：抽出一个通用的"生成任务"抽象（创建/轮询/流式/取消/重试），所有 AI 场景通过策略模式接入同一个任务引擎；把 `external_ai_call_logs` 升级为真正的成本与质量仪表盘，记录延迟、成本、主观质量打分。
- **理由（Why）**：现在每加一个新 AI 场景都要重新写一遍任务状态机，收敛后新场景接入成本会大幅下降；同时"要不要换个模型/供应商"这种决策现在完全凭感觉，有了数据仪表盘才能做真正的成本-质量权衡。
- **验收标准**：新增一个 AI 场景时，只需要实现"输入->调用->输出解析"这一小段策略代码，不需要重新写任务状态机、轮询、重试逻辑；仪表盘能展示至少"各场景近30天的调用次数、平均延迟、失败率、粗略成本"。

## 五、数据层

### [P2] 版本历史与软删除的通用化
- **现状**：只有宫殿脑图（`PalaceVersion`）有"版本快照+去重合并+里程碑强制快照"机制；其余实体是硬删除。
- **目标**：把这套模式泛化成所有内容实体的通用能力（配合功能层清单的"全局撤销/回收站"）。
- **理由（Why）**：宫殿脑图和其他实体（题目、章节、附件）没有本质区别，值得"错删了能找回"这个安全网的实体不该只有一个。
- **验收标准**：至少题目、章节、迷你宫殿三类高频编辑实体具备软删除+一段时间内可恢复的能力。

## 六、前端架构

### [P2] 脑图编辑器从"iframe + 关闭全站 HMR"改为独立微前端
- **现状**：思维导图编辑器用 iframe host 隔离，因为对热更新不稳定，dev 模式因此**关闭了整个应用的 HMR**，代价是每次改任何前端代码都要手动刷新整站。
- **目标**：把脑图编辑器拆成一个有独立 dev server 的微前端，主应用通过稳定契约消费它，恢复主应用其余部分的正常 HMR。
- **理由（Why）**：为了一个编辑器的稳定性牺牲全站的开发反馈速度，长期看是不成比例的代价，尤其你是唯一开发者，开发效率直接影响这个项目能走多远。
- **验收标准**：修改非脑图相关的前端代码时 HMR 正常工作，无需整页刷新；脑图编辑器本身的稳定性不受影响。

### [P3] 确认弹窗方案收敛
- **现状**：`native-dialog.tsx` 的 `appConfirm`/`appAlert` 已经是主流方案（13 处引用），但仍残留一个平行的 `confirm-dialog.tsx`（仅 1 处引用，`PalaceQuizPage.tsx`）。
- **目标**：把最后这一处迁移到统一方案，删除平行组件。
- **理由（Why）**：两套弹窗方案并存意味着同样的删除操作在不同页面长得不一样，这是最容易造成直觉疑惑的根源之一，且改动成本很低（只有 1 处引用）。
- **验收标准**：`confirm-dialog.tsx` 被删除，`PalaceQuizPage.tsx` 改用 `appConfirm`。

## 七、测试与可维护性

### [P1] 把测试投入重心转移到"一旦出错就是数据永久丢失"的模块
- **现状**：前端 99 个测试文件集中在计时器/反馈/脑图宿主上；后端 24 个测试文件覆盖相对全仓库模块体量偏薄，尤其 backup/restore、file_sync 这类模块测试不够扎实。
- **目标**：优先为 `backups`、`file_sync`（同步）、数据库迁移路径补充端到端测试，覆盖"备份恢复后数据完整""同步冲突场景不丢数据""迁移失败可回滚"这几类高风险场景。
- **理由（Why）**：这是个人单用户产品，没有别的用户帮你踩坑，一旦这几个模块出错，代价是真实学习数据的永久损失，投入优先级应该高于继续给反馈动效加测试。
- **验收标准**：`backups`、`file_sync` 模块拥有专门的测试文件，覆盖"正常备份-恢复""并发写入冲突""恢复中途失败"等场景；这些测试成为 CI 的强制检查项。
