# 方向claude

> 原始路径：大局\方向claude.md

基于对代码库的完整审视（模块清单、数据模型、同步机制、AI 工作流、反馈系统、测试覆盖、技术债），我给出一份不受当前实现束缚的长远产品与架构规划。先说一个前提判断，再展开三层。

## 定位判断：先解决"身份分裂"问题

代码库现在同时是三个产品的叠加：①通用的"内容→记忆宫殿→题库→间隔重复"引擎（palaces/knowledge/palace_quiz/reviews）；②一个独立的英语学习产品（english 听力 + english_reading 阅读，各自重新实现了 session、progress、生成任务、校准逻辑）；③一堆围绕专注计时/情绪反馈的桌面小工具（timer、voice_coach、desktop-timer）。这三者目前是并列的烟囱，而不是同一引擎的三个应用层。

**长远愿景应该是收敛成一个东西**：一个通用的"任意学习材料 → AI 结构化 → 科学复习"引擎，英语听力/阅读不再是独立模块，而是这个引擎上的两个"内容适配器"（音视频源、分级文本源），复用同一套 session/progress/AI-job/复习调度基础设施。这个决定会连带影响下面几乎所有架构建议。

---

## 一、功能层

**该砍/降级**
- `english` / `english_reading` 不删除，但降级为"内容适配器"，停止各自维护平行的 session/progress/generation-task 实现，改为消费统一的会话与生成任务基础设施。
- `freestyle` 目前只有 3 个 application 文件，体量和产品意图都不清晰——要么在这轮规划里给它一个明确定位（比如"错题本/薄弱点自由练习流"，见下），要么砍掉，不要让它长期停留在"半成品"状态。
- 时间记录的 legacy 迁移脚手架（`session-records-legacy.ts`、`import-legacy` 端点、`normalize-timezones`）：既然是个人产品、双设备都在你控制下，一旦确认线上数据已完成迁移，这类兼容层应该有计划地清场，而不是永久挂着。
- 仓库里 `.audit/`、`merge_1000_*.py` 这类一次性生产脚本产物，属于纯历史遗留，不影响功能但影响你对代码库"干净度"的信任感，建议归档到仓库外或彻底移除。
- 四个 AI 供应商目前是"各管一摊"（谁配置了谁用），应收敛为下面架构层说的统一网关，供应商本身可以保留，但不该是各场景硬编码的选择。

**该优化**
- **统一错题/薄弱点体系**：现在测验的错题、复习中反复"翻车"的节点、阅读校准中的失误，是三套割裂的数据。应该收敛成一个跨内容类型的"薄弱点流"，作为一个独立的高价值 Feed，而不是分散在各模块里被动等待复习轮次。这是个人 SRS 产品里投入产出比最高的功能。
- **复习判分反馈的不对称**：宫殿复习是"揭示式"，天然没有对/错判定，反馈系统目前也确实只做正向强化（连击、里程碑、彩带）；但测验模块有真实对错，这里的即时反馈（尤其选择题）应该做到比 toast/彩带更快、更轻的"毫秒级"视觉反馈——这两类反馈的设计逻辑应该被明确区分对待，而不是共用一套"庆祝引擎"心智。
- **同步体验**：当前"整包快照 + 冲突即阻断"的模式，本质要求你严格串行使用两台设备。这是当前最大的隐性风险点——只要有一次忘记同步就手动关机，就会触发冲突快照，需要人工介入。见架构层的具体方案。

**该新增**
- **统一薄弱点/错题本**（如上）。
- **遗忘曲线与"高危知识点"预测仪表盘**：现有 dashboard 只做"今日复习量、时长统计"这类回顾型指标；结合下面提到的 FSRS 调度算法，完全可以做"哪些节点大概率会被遗忘""哪些主题保留率在下降"这类前瞻型分析，这才是间隔重复系统真正的杀手级体验。
- **AI 学习教练升级为可对话式**：voice_coach 目前只是场景化 TTS 播报器；既然已经接入了通用 LLM 能力，应该做成"针对当前错题/节点提问、要求换一种讲法、临时生成一张类比卡片"的轻量对话入口，而不是单向播报。
- **全局撤销/回收站**：目前只有宫殿脑图有版本历史（`PalaceVersion`），其余删除操作（测验题、章节、附件）都是硬删除。应该把这套"软删除 + 可回收"的能力，从宫殿脑图这一个实体，泛化成所有可删除实体的默认行为。
- **全局速记捕获**：一个不打断心流的快捷键，随时记下一个念头/一道临时想到的题/一个疑问，进入收件箱，稍后再分诊进宫殿或题库——现在没有这种"低摩擦捕获"入口，学习中的灵光一现容易在专注计时器和当前任务里被压掉。
- **语料的语义检索**：既然已经在积累 PDF、笔记、思维导图等结构化内容，统一内容摄入管道天然可以顺带做一个"对自己知识库提问"的检索层（不需要复杂 RAG，SQLite FTS5 + 简单向量索引就够用）。

## 二、架构层

**前端**：FSD 分层本身是合理的，保留。但要把 english/english_reading 的平行实现，收敛进通用的 `entities/learning-session`、`entities/generation-job` 这类跨垂直的抽象，而不是每个垂直都自己重新发明 session/progress/AI 任务轮询逻辑。思维导图编辑器目前用 iframe host 隔离、并且**为此关闭了整个应用的 dev HMR**——这是一个明显不成比例的代价（为了一个编辑器牺牲了全站的开发反馈速度）。长远应该把脑图编辑器拆成一个独立的、有自己独立 dev server 的微前端，主应用通过稳定契约消费它，而不是用 iframe + 关 HMR 这种"绕开问题而不是解决问题"的做法。

**后端**：`docs/architecture/boundary-exceptions.json` 里已经登记了 4 条"临时"跨模块依赖（reviews↔palaces、backups↔mindmap），这些债务是有明确移除条件的，应该被当作一个正式的技术债项目排期清偿，而不是永久停留在"临时"状态——反复展期的临时豁免就是事实上的永久债务。`palace_quiz` 被拆成约 150 个小文件，是过度拆分的信号，值得按内聚的命令簇重新归并，降低认知负担。

**调度算法**：当前是艾宾浩斯/SM2 混合调度。长远建议迁移到 **FSRS**（Free Spaced Repetition Scheduler）——这是目前公认优于 SM2/朴素艾宾浩斯的开源调度算法，能根据你真实的复习历史反推出个性化的遗忘曲线参数，而不是用固定间隔表。这也是"高反馈"承诺的算法基础：调度越准，"该复习的时候恰好推给你"这件事的信噪比就越高。

**AI 工作流**：现在是"每个场景硬编码调用哪家供应商、什么模型"。应该收敛成一个通用的 **AI 网关层**：统一的"生成任务"抽象（创建/轮询/流式/取消/重试），所有垂直场景（思维导图导入、测验生成、ASR、CEFR 分级）通过策略模式接入同一个任务引擎，而不是各自维护一套 Job 表和轮询逻辑。已经有的 `external_ai_call_logs` 应该从"调用日志"升级为真正的"AI 成本与质量仪表盘"——记录每个场景的延迟、成本、你主观打分的生成质量，让"要不要换个模型"这种决策有数据支撑而不是凭感觉。

**数据层**：把 `PalaceVersion` 的"版本快照+去重合并+里程碑强制快照"模式，泛化成所有内容实体的通用 revision 机制，而不是宫殿脑图独有。软删除（`deleted_at` + TTL 清理）应该是默认设计，而不是特例。

**同步架构（这是整个规划里我认为分量最重的一项）**：当前"snapshot 整包同步 + 冲突阻断"的方案，本质上是用"假装单机"的方式规避真正的多设备问题——能用，但脆弱，且随着数据量增大（附件、题库、脑图快照）同步包会越来越重。长远两个可选方向，按投入递增排序：
1. **轻量改良**：保持整包快照思路，但把"冲突即阻断、需要人工翻文件夹"升级为一个应用内的"合并向导"UI，让冲突处理不再是文件系统层面的手工活。
2. **架构级重做**（真正有魄力的选择）：把 SQLite 的写操作视为一条可重放的操作日志（不是文件级 diff，而是"发生了什么"事件流），两台设备互相拉取、按行/按实体做真正的合并而不是整包覆盖。可以借助 SQLite 的 WAL/变更捕获机制自建一个极简的个人同步服务（哪怕只是一台常开的小型 VPS 或者树莓派跑一个 FastAPI 同步端点），彻底告别"必须严格串行使用两台设备"的约束，这也是唯一能真正兑现"两台电脑无缝切换、随时用"这个产品承诺的路径。

**测试与可维护性**：当前 99 个前端测试集中在计时器/反馈/脑图宿主上，后端 24 个测试文件相对模块体量偏薄，而恰恰是 backup/restore、sync 这些"一旦出错就是数据永久丢失"的模块测试覆盖不够扎实。对个人单用户产品而言，没有别的用户能帮你踩坑，这部分应该是投入优先级最高的测试领域，而不是继续在反馈动效上加测试。

## 三、体验层

**视觉/听觉反馈的两层区分**：现有的彩带庆祝、连击里程碑、TTS 播报体系已经相当完善，但它服务的是"宏观里程碑"反馈。真正影响心流的是"微观、瞬时"反馈——比如测验答对/答错后 200ms 内的颜色/图标响应，这类反馈应该做到安静、极快、不打断节奏，和里程碑级的"大庆祝"在设计语言上明确区分，不要用同一套彩带引擎覆盖所有场景。

**心流模式**：建议增加一个"沉浸模式"UI 状态——连续正确 N 次后自动收起多余的界面元素，只留题目/卡片居中、纯键盘操作、计时器降到极简存在感，用"减少界面摩擦"而不是"更多庆祝特效"来奖励进入状态的用户，这比堆叠更多动效更符合"易于进入心流"的目标。

**断点续跑**：这块目前做得扎实——`SessionProgress` 按 session_kind 分类、每次操作同步 commit 落盘，断电/意外关闭基本能恢复到中断点。长远只需要把这个保证延伸到 AI 生成任务上（比如 PDF 转题库中途中断，应该能从断点续跑，而不是重新生成）。

**进度保存与意外情况的可见性**：目前最大的体验盲区是 `mutationQueue` 离线重试队列对用户完全不可见——保存失败后用户没有任何"你的修改还没同步成功"的提示，只能靠刷新页面赌运气。应该增加一个持久的小状态指示（已同步/保存中/离线排队/冲突待处理），把这个隐性风险变成可感知、可信任的状态。同时把回收站/软删除机制铺开到所有实体，让你可以放心大胆地删改，而不用担心手滑。

**减少直觉疑惑**：代码里已经有统一的确认弹窗方案（`native-dialog.tsx` 的 `appConfirm`/`appAlert`，13 处引用），但还残留一个平行的 `confirm-dialog.tsx`（仅 1 处引用）——这种"两套方案并存"正是最容易造成用户直觉疑惑的根源（同样的删除操作，弹窗长得不一样），应该尽快收敛成一套，并把"什么场景用 toast、什么场景用弹窗、什么场景用屏幕反馈"写成一份明确的内部规范，让未来新功能默认遵循，而不是每个功能各自发明一套交互。
