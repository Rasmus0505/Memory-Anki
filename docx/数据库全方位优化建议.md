
技术背景：SQLite WAL + SQLAlchemy 2.0 同步 Session + FastAPI。以下按影响程度从高到低排列，每条建议都指向具体代码位置。

---

### 一、查询性能：高影响问题

**1. Dashboard N+1 查询：最严重的性能瓶颈**

[dashboard/application/service.py:84](app://localhost/epitaxy/apps/api/src/memory_anki/modules/dashboard/application/service.py)存在一个经典N+1：

```python
all_palaces = list_palaces(session)          # 拉取全部宫殿 + 所有关联
for palace in all_palaces:
    counts = count_palace_review_units(session, palace)  # 每个宫殿再执行逻辑
```

`count_palace_review_units` 内部又调用 `palace_has_due_review` → `is_schedule_due`，该函数可能再次查询 Config 表。改为单次聚合查询：

```python
# 用一条SQL 统计 due_now / due_later / needs_practice，而不是逐个宫殿循环
from sqlalchemy import func, case

due_stats = (
    session.query(
        func.sum(case((ReviewSchedule.scheduled_date <= today, 1), else_=0)).label("due_now"),
        ...
    )
    .filter(ReviewSchedule.completed == False)
    .scalar()
)
```

**2. `restore_archived_palaces` 在同一请求中被调用多次**

`list_palaces`、`get_palace`、`list_catalog_palaces`、`_due_query` 每个都调用一次 `restore_archived_palaces`，后者执行一个 `UPDATE`。Dashboard 里一次请求调用了 `list_palaces` 和 `get_today_review_groups`，等于两次无意义的重复写操作。

建议在 FastAPI 的 request 生命周期内用一个 flag 控制，或者把这个逻辑移到启动时（实际上 archived宫殿几乎不会累积）。

**3. `_due_query` 把过滤逻辑放在 Python 层**

[review_queue_service.py:27](app://localhost/epitaxy/apps/api/src/memory_anki/modules/reviews/application/review_queue_service.py) 拉取所有未完成的 schedule，再在 Python 里逐个判断 `is_schedule_due`：

```python
schedules = query.all()               # 全量拉取
return [s for s in schedules if is_schedule_due(s, ...)]  # Python 过滤
```

`is_schedule_due` 内部逻辑基本是日期比较，应该直接下推到 SQL `WHERE scheduled_date <= :today AND (scheduled_at IS NULL OR scheduled_at <= :now)` 里，避免把不需要的行传输到内存。

### 三、SQLite PRAGMA 调优

[infrastructure/db/_tables/_base.py:27](app://localhost/epitaxy/apps/api/src/memory_anki/infrastructure/db/_tables/_base.py) 目前的PRAGMA 配置基础良好，还可以补充以下几项：

```python
@event.listens_for(engine, "connect")
def _configure_sqlite_pragmas(dbapi_connection, _connection_record) -> None:
    if not isinstance(dbapi_connection, sqlite3.Connection):
        return
    cursor = dbapi_connection.cursor()
    try:
        cursor.execute("PRAGMA foreign_keys=ON")
        cursor.execute("PRAGMA busy_timeout=30000")
        cursor.execute("PRAGMA synchronous=NORMAL")
        cursor.execute("PRAGMA journal_mode=WAL")# 新增：增大内存缓存（默认 -2000 ≈ 2MB，改为 ~32MB）
        cursor.execute("PRAGMA cache_size=-32000")

        # 新增：temp表用内存，减少临时磁盘 I/O
        cursor.execute("PRAGMA temp_store=MEMORY")

        # 新增：Windows 下 mmap 提升读性能（256MB）
        cursor.execute("PRAGMA mmap_size=268435456")
    finally:
        cursor.close()
```

`synchronous=NORMAL` 已经很好（比 FULL 快，WAL 模式下数据安全有保障）。`cache_size` 当前是默认 2MB，对于包含 `editor_doc`（大 JSON）的宫殿表来说太小，会频繁换页。

### 四、ORM 加载策略：过度 eager loading

**4.1 `lazy="joined"` 带来隐式 JOIN**

[palaces.py:131](app://localhost/epitaxy/apps/api/src/memory_anki/infrastructure/db/_tables/palaces.py) 的 `primary_chapter` 关系定义了 `lazy="joined"`：

```python
primary_chapter: Mapped[Chapter | None] = relationship(
    "Chapter", foreign_keys=[primary_chapter_id], lazy="joined"  # 每次查 Palace 都JOIN
)
```

这意味着任何时候 ORM 加载 `Palace`（包括只需要 id/title 的场景），都会触发对 `chapters` 的 JOIN。同样的问题存在于 `PalaceQuizQuestion` 的 `source_chapter`（`lazy="joined"`）和 `classified_chapter`（`lazy="joined"`）。

建议将这些改为 `lazy="select"`（默认值），在需要时用 `joinedload()`/`selectinload()` 显式控制。

**4.2 `_detail_loader_options` 即使不需要 pegs 时也加载**

`list_palaces` 用的是 `_detail_loader_options()`，里面包含 `selectinload(Palace.pegs)`。宫殿列表页其实不需要 peg 内容，改用 `_catalog_loader_options()` 即可。

### 五、数据增长：需要制定清理策略的表

以下表没有自动清理机制，随时间会持续增长并拖慢 VACUUM：

**5.1 `palace_versions`（版本快照）**

每次手动保存都写一行，`editor_doc`、`editor_config`、`peg_snapshot` 都是大 Text 字段。建议保留最近N 个版本（如 20 个），定期删除旧版本：

```python
# 在 backup 或 scheduled task 里执行
from sqlalchemy import func
subq = (
    session.query(PalaceVersion.id)
    .filter(PalaceVersion.palace_id == palace_id)
    .order_by(PalaceVersion.created_at.desc())
    .limit(20)
    .subquery()
)
session.query(PalaceVersion).filter(
    PalaceVersion.palace_id == palace_id,
    ~PalaceVersion.id.in_(subq),
).delete(synchronize_session=False)
```

**5.2 `external_ai_call_logs`（AI 调用日志）**

`request_json`/`response_json` 存储完整的 AI 请求和响应，每次AI 调用都会新增一行大文本。建议：

- 超过 90 天的日志定期删除
- 或者把 `request_json`/`response_json` 分离到文件系统（已有 `.audit/` 目录），数据库只保留元数据和 status

**5.3 `study_sessions` 的 `events_json`**

`events_json` 字段存储了完整的会话事件流（鼠标点击、进度更新等），对于已结束的会话，这个字段可以在归档后清空或截断，只保留 `summary_json`。

### 六、定期维护：VACUUM 和 ANALYZE

当前启动流程没有触发 SQLite 的统计更新。在宫殿数量或版本快照大量增长后，查询计划可能因统计过时而走错索引。

建议在 `backup_service.py` 已有的定期备份逻辑中，同步触发：

```python
from sqlalchemy import text

def run_periodic_db_maintenance(session: Session) -> None:
    """在备份之后运行，不阻塞正常请求。"""
    # 更新查询计划统计
    session.execute(text("ANALYZE"))
    session.commit()
    # 每周一次 VACUUM（在备份后执行，不影响备份完整性）
    session.execute(text("PRAGMAwal_checkpoint(TRUNCATE)"))
```

注意 `VACUUM` 是全库操作，建议只在关闭时（`stop.bat` 调用时）或低频触发，而 `ANALYZE` 可以放在每次备份后。

### 七、连接池配置

[_base.py:20](app://localhost/epitaxy/apps/api/src/memory_anki/infrastructure/db/_tables/_base.py) 使用默认配置：

```python
engine = create_engine(
    DATABASE_URL,
    connect_args={"check_same_thread": False, "timeout": 30},
)
```

SQLAlchemy 对SQLite 默认使用 `StaticPool`（单连接），这对单用户本地应用是合适的，但由于 FastAPI 是多线程的（Uvicorn 默认多worker），`check_same_thread=False` 已经放开了线程限制。可以显式配置连接池行为来避免歧义：

```python
from sqlalchemy.pool import StaticPool

engine = create_engine(
    DATABASE_URL,
    connect_args={"check_same_thread": False, "timeout": 30},
    poolclass=StaticPool,# 明确：单连接，适合 SQLite WAL 单进程场景
)
```

如果后续想支持更高并发，可以改为 `NullPool`（每次请求独立连接），搭配 WAL 模式的多读者特性。

### 八、`chapter_palaces` 关联表缺少唯一约束

[palaces.py:39](app://localhost/epitaxy/apps/api/src/memory_anki/infrastructure/db/_tables/palaces.py) 的 `chapter_palace_table` 没有 `(chapter_id, palace_id)` 的唯一约束，理论上可以插入重复关联行：

```python
chapter_palace_table = Table(
    "chapter_palaces",
    Base.metadata,
    Column("id", Integer, primary_key=True, autoincrement=True),
    Column("chapter_id", Integer, ForeignKey("chapters.id", ondelete="CASCADE")),
    Column("palace_id", Integer, ForeignKey("palaces.id", ondelete="CASCADE")),
    Column("is_explicit", Boolean, nullable=False, default=True),
    UniqueConstraint("chapter_id", "palace_id", name="uq_chapter_palaces"),# 加上这行
)
```
