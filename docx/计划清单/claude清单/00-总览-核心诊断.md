# Memory Anki 产品架构愿景 - 总览

> 撰写日期：2026-07-03  
> 审视角色：资深产品架构师 + 产品设计师  
> 项目性质：个人自用本地学习产品，双设备使用，不面向公众

---

## 🎯 核心问题诊断

### 1. 定位模糊

**现状**：
- 试图做"记忆宫殿 + 间隔重复 + 题库 + 英语听力 + 英语阅读 + 知识体系"
- 13个业务模块，功能堆砌严重
- 各功能域之间缺乏有机联系

**根本问题**：
- 缺乏清晰的核心价值主张
- 到底是要解决"高效记忆"还是"知识管理"还是"学习训练"？

**应该成为**：
> **Memory Anki = 记忆宫殿驱动的智能学习系统**
> 
> 核心价值：将任何学习材料转化为"可视化记忆宫殿"，通过科学的间隔重复和即时反馈，帮助用户高效记忆、低负担学习、轻松进入心流状态。

---

### 2. 过度工程化

**现状**：
- 后端：13个模块 + DDD分层 + 跨模块契约 + 架构检查脚本
- 前端：FSD分层 + 路由LRU缓存 + 离线变更队列 + iframe隔离
- 技术栈：React 19 + Vite 8 + Python 3.12+ + FastAPI + SQLAlchemy 2.0

**根本问题**：
- 企业级架构应用于个人项目，维护成本极高
- 复杂度与价值不匹配
- 技术选型激进，但未带来对应的体验提升

**应该做到**：
> **极简架构，最大价值**
> 
> - 后端：models + services + api，去除DDD分层
> - 前端：pages + components + hooks，去除FSD抽象
> - 数据：扁平化表结构 + JSON灵活存储
> - 测试：50%覆盖率即可，专注核心逻辑

---

### 3. 用户体验断层

**现状**：
- 功能完备，但学习曲线陡峭
- 反馈系统复杂（音频坐标器、庆祝引擎、全局反馈模型）
- 缺乏情感连接和成就感

**根本问题**：
- 把"工具"而非"体验"放在首位
- 用户需要理解"Peg/Segment/MiniPalace/ReviewSchedule"等技术概念
- 缺乏主动式引导和学习陪伴感

**应该做到**：
> **沉浸式学习体验**
> 
> - 去概念化：用"知识点/组/关卡"替代技术术语
> - 视觉优先：记忆强度热力图、遗忘风险预警、成长曲线
> - 主动引导：每日推荐、智能计划、里程碑奖励
> - 即时反馈：编辑即练习、答题即标注、学习即成长

## ⚠️ 重要原则

在后续执行中，请牢记：

1. **删除优先于新增**：先做减法，再做加法
2. **体验优先于功能**：宁可功能少，也要体验好
3. **简单优先于完美**：个人项目，灵活性比严格性重要
4. **可见优先于隐藏**：让学习进度和成就可视化
5. **渐进优先于重写**：分阶段重构，保持可用状态
